    # MySQL - 数据类型

    =============================

    本文主要整理MySQL中数据字段类型。@pdai

    # 字段类型

    # 整型

    TINYINT, SMALLINT, MEDIUMINT, INT, BIGINT 分别使用 8, 16, 24, 32, 64 位存储空间，一般情况下越小的列越好。

    类型 位数 字节 有符号范围 最大数字位数 无符号范围 最大数字位数
    TINYINT 8 1 -128 ~ 127 3 位 0 ~ 255 3 位
    SMALLINT 16 2 -32768 ~ 32767 5 位 0 ~ 65535 5 位
    MEDIUMINT 24 3 -8388608 ~ 8388607 7 位 0 ~ 16777215 8 位
    INT 32 4 -2147483648 ~ 2147483647 10 位 0 ~ 4294967295 10 位
    BIGINT 64 8 -9223372036854775808 ~ 9223372036854775807 19 位 0 ~ 18446744073709551615 20 位

    INT(11)中的数字只是规定了交互工具显示字符的个数，对于存储和计算来说是没有意义的。

    ps ：
    1 Byte（字节） = 8 bit（位）
    1 KB = 1024 Byte
    1 KB = 1024 × 8 = 8192 bit

    # 浮点数

    FLOAT 和 DOUBLE 为浮点类型，DECIMAL 为高精度小数类型。CPU 原生支持浮点运算，但是不支持 DECIMAl 类型的计算，因此 DECIMAL 的计算比浮点类型需要更高的代价。

    FLOAT、DOUBLE 和 DECIMAL 都可以指定列宽，例如 DECIMAL(18, 9) 表示总共 18 位，取 9 位存储小数部分，剩下 9 位存储整数部分。

    存储钱币时 DOUBLE 不如 DECIMAL 合适，核心原因是：DOUBLE（浮点型）存在二进制浮点精度丢失问题，会导致金额计算出错；而 DECIMAL（定点型）是十进制精准存储，完全避免这类误差。

    ps: MySQL 对 DECIMAL 类型的核心限制规则：

    参数 限制值 说明
    M（总位数） 1 ~ 65 官方明确的最大值为 65，超过会报错（如 DECIMAL(66,30) 直接创建失败）
    D（小数位） 0 ~ M 小数位数不能超过总位数，且实际业务中 MySQL 建议 D 不超过 30（精度溢出风险）

    DECIMAL(65,30) 是 M 取最大值 65、D 取常用最大值 30 的组合，是 DECIMAL 能定义的最大位数规格；

    # 字符串

    主要有 CHAR 和 VARCHAR 两种类型，一种是定长的，一种是变长的。

    VARCHAR 这种变长类型能够节省空间，因为只需要存储必要的内容。但是在执行 UPDATE 时可能会使行变得比原来长，当超出一个页所能容纳的大小时，就要执行额外的操作。MyISAM 会将行拆成不同的片段存储，而 InnoDB 则需要分裂页来使行放进页内。

    VARCHAR 会保留字符串末尾的空格，而 CHAR 会删除。

    # 时间和日期

    MySQL 提供了两种相似的日期时间类型: DATETIME 和 TIMESTAMP。

    # DATETIME

    能够保存从 1001 年到 9999 年的日期和时间，精度为秒，使用 8 字节的存储空间。

    它与时区无关。

    默认情况下，MySQL 以一种可排序的、无歧义的格式显示 DATETIME 值，例如“2008-01-16 22:37:08”，这是 ANSI 标准定义的日期和时间表示方法。

    # TIMESTAMP

    和 UNIX 时间戳相同，保存从 1970 年 1 月 1 日午夜(格林威治时间)以来的秒数，使用 4 个字节，只能表示从 1970 年 到 2038 年。

    它和时区有关，也就是说一个时间戳在不同的时区所代表的具体时间是不同的。

    MySQL 提供了 FROM_UNIXTIME() 函数把 UNIX 时间戳转换为日期，并提供了 UNIX_TIMESTAMP() 函数把日期转换为 UNIX 时间戳。

    默认情况下，如果插入时没有指定 TIMESTAMP 列的值，会将这个值设置为当前时间。

    应该尽量使用 TIMESTAMP，因为它比 DATETIME 空间效率更高。

    # 选择优化的数据类型

    • 更小的通常更好；更小的数据类型通常更快，因为它们占用更少的磁盘、内存和CPU缓存，并且处理时需要的CPU周期也更少；
    • 简单就好；例如，整形比字符串操作代价更低；实用内建类型而不是字符串来存储日期和时间；用整形存储IP地址等；
    • 尽量避免NULL；如果查询中包含可为NULL的列，对MySQL来说更难优化，因为可为NULL 的列使得索引、索引统计和值比较都更复杂。尽管把可为NULL的列改为NOT NULL带来的性能提升比较小，但如果计划在列上创建索引，就应该尽量避免设计成可为NULL的列；

    # 字符串类型

    # VARCHAR 和 CHAR

    对比项 CHAR(M) VARCHAR(M)
    长度类型 固定长度 可变长度
    末尾空格 存储时自动删除 原样保留
    比较行为 固定按定义的 M 长度补空格（超长查询条件先截断到 M 位） 按「字段值 / 查询条件中较长者」补空格（无截断，仅临时补）
    存储空间 固定：M × 字符编码单字节数（如 utf8mb4 下 CHAR (10) 占 40 字节） 可变：实际字符数 × 单字节数 + 1~2 字节（如 utf8mb4 下存「张三」占 6+1=7 字节）
    性能 更快（固定长度，磁盘 / 内存定位效率高） 略慢（需计算长度，碎片略多）
    适合场景 固定长度、短字段（性别、状态） 不固定长度（姓名、标题、内容）

    VARCHAR是最常见的字符串类型。VARCHAR节省了存储空间，所以对性能也有帮助。但是，由于行是可变的，在UPDATE时可能使行变得比原来更长，这就导致需要做额外的工作。如果一个行占用的空间增长，并且在页内没有更多的空间可以存储，MyISAM会将行拆成不同的片段存储(行分片存储（Row Fragmentation）)；InnoDB则需要分裂页来使行可以放进页内(页分裂)。

    # MyISAM：行分片存储（Row Fragmentation）

    • MyISAM 每行数据在物理上是连续的一段内存 / 磁盘块 ，当你 UPDATE 让一行变长，原来位置放不下了 。MyISAM 不会移动整行，而是：把新增长的部分切出去，存到别的空闲位置，原行只保留指针指向新碎片
    • 这就是行分片（行碎片）。

    # InnoDB：页分裂（Page Split）

    • InnoDB 是索引组织表，数据存在 B+ 树的叶子节点里。最小存储单元：16KB 页（Page） ，页必须整体连续、有序 一页满了就不能再塞数据 。当某行 UPDATE 变长，导致当前页放不下：
    • InnoDB 不能像 MyISAM 那样拆行 ，必须把当前页劈成 2 页 ，把部分数据挪到新页 ，调整 B+ 树指针
    • 这就叫页分裂。
    对比维度 分片存储（MyISAM） 页分裂（InnoDB）
    发生引擎 MyISAM 引擎 InnoDB 引擎
    本质对象 对单行数据进行拆分 对 ** 数据页（16KB）** 进行拆分
    触发原因 行更新后变长，原位置放不下，拆成多段碎片 行更新 / 插入后变长，当前页满，无法容纳新数据
    存储结构 一行数据被拆成多个碎片，用指针串联分散存储 一个完整数据页被劈成 2 页，部分数据迁移到新页
    数据连续性 行数据物理不连续，存在碎片 页内数据仍保持有序、连续，只是页数量增加
    IO 影响 读取一行需多次随机 IO，拼接碎片，性能下降 分裂时产生额外 IO 与树结构调整，写入卡顿
    空间利用 产生大量空洞，空间浪费 页内仍有预留空间，分裂后整体空间利用率略降
    优化手段 OPTIMIZE TABLE 重组表 / 迁移到 InnoDB 控制字段长度、使用自增主键、定期 ALTER TABLE ... ENGINE=InnoDB 重建表
    底层数据结构 堆表结构，行物理存储位置独立 索引组织表（B + 树），数据页是叶子节点

    下面这些情况使用VARCHAR是合适的：字符串的最大长度比平均长度大很多；列的更新很少，所以碎片不是问题；使用了像UTF-8这样复杂的字符集，每个字符都使用不同的字节数进行存储。

    当存储CHAR值时，MySQL会删除所有的末尾空格。CHAR值会根据需要采用空格进行填充以方便比较。

    CHAR适合存储很短的字符串，或者所有值都接近同一个长度，如密码的MD5值。对于经常变更的数据，CHAR也比VARCHAR更好，因为CHAR不容易产生碎片（行间碎片？）。

    ps: 不同字符在 utf8mb4 编码下的字节长度

    字符类型 示例 字节长度 补充说明
    基本 ASCII 字符（数字 / 字母 / 符号） 0-9、a-z、A-Z、!@#$ 1 字节 所有 ASCII 码表内的字符（0-127）均占 1 字节
    常见中文（简体 / 繁体） 男、张、三、测、试 3 字节 绝大多数常用中文（GBK 覆盖的）均占 3 字节
    生僻中文 / 特殊汉字 𰻝、𬎆（扩展区汉字） 4 字节 Unicode 扩展区的生僻字，需 utf8mb4 才能存储
    Emoji 表情 😊、🚗、🔥、🎉 4 字节 所有 Emoji 均占 4 字节，utf8（3 字节）无法存储
    日韩文字 あ、か、韓、日 3 字节 常用日文假名、韩文汉字均占 3 字节
    空格 / 制表符 （空格）、\t 1 字节 普通空格、制表符、换行符均为 1 字节
    特殊符号（非 ASCII） €、£、§ 3 字节 欧洲货币符号、特殊符号等占 3 字节
    组合字符（如带声调） é、ñ、à 2 字节 基础字符 + 声调符号的组合，占 2 字节

    # VARCHAR(5)和VARCHAR(200)

    使用VARCHAR(5)和VARCHAR(200)存储"hello"的空间开销是一样的。那么使用更短的列有什么优势吗？

    事实证明有很大的优势。更长的列会消耗更多的内存，因为MySQL通常会分配固定大小的内存块来保存内部值。尤其是使用内存临时表进行排序或其他操作时会特别糟糕。在利用磁盘临时表进行排序时也同样糟糕。

    所以最好的策略是只分配真正需要的空间。

    # BLOB 和 TEXT

    BLOB和TEXT都是为存储很大的数据而设计的数据类型，分别采用二进制和字符方式存储。

    与其他类型不同，MySQL把每个BLOB和TEXT值当做一个独立的对象去处理。当BLOB和TEXT值太大时，InnoDB会使用专门的“外部”存储区域来进行存储，此时每个值在行内需要1~4个字节存储一个指针，然后在外部存储区域存储实际的值。

    MySQL对BLOB和TEXT列进行排序与其他类型是不同的：它只对每个列的最前max_sort_length个字节而不是整个字符串做排序。同样的，MySQL也不能将BLOB或TEXT列全部长度的字符串进行索引。

    # 选择表示符（identifier）

    整数类型通常是标识列的最佳选择，因为它们很快并且可以使用AUTO_INCREMENT。 如果可能，应该避免使用字符串类型作为标识列，因为它们很耗空间，并且比数字类型慢。 对于完全随机的字符串也需要多加注意，例如MD5(),SHA1()或者UUID()产生的字符串。这些函数生成的新值会任意分布在很大的空间内，这会导致INSERT以及一些SELECT语句变得很慢：

    • 因为插入值会随机的写入到索引的不同位置，所以使得INSERT语句更慢。这会导致页分裂、磁盘随机访问。
    • SELECT语句会变的更慢，因为逻辑上相邻的行会分布在磁盘和内存的不同地方。
    • 随机值导致缓存对所有类型的查询语句效果都很差，因为会使得缓存赖以工作的局部性原理失效。)
