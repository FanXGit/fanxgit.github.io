    # MySQL - 索引

    ==================================

    https://www.cnblogs.com/xiaoxi/p/6894610.html

    # 索引的优点

    • 大幅提高查询速度 :避免全表扫描，通过 B+ 树快速定位数据，百万、千万级数据提升尤其明显。
    • 降低服务器 CPU、IO 消耗 ：不用逐行扫描，减少磁盘随机 IO 和 CPU 计算。也就不需要创建临时表(B+Tree 索引是有序的，可以用于 ORDER BY 和 GROUP BY 操作。临时表主要是在排序和分组过程中创建，因为不需要排序和分组，也就不需要创建临时表)。
    • 提升排序、分组效率 ：ORDER BY、GROUP BY、DISTINCT 如果命中索引，可以直接用索引有序结构，避免额外排序。也就不需要创建临时表(B+Tree 索引是有序的，可以用于 ORDER BY 和 GROUP BY 操作。临时表主要是在排序和分组过程中创建，因为不需要排序和分组，也就不需要创建临时表)。
    • 加速表连接（JOIN） ：关联字段建索引，能显著提升多表联查速度。
    • 可以实现唯一性约束 ：唯一索引（UNIQUE）保证字段不重复，比程序校验更可靠。
    • 覆盖索引可以避免回表 ：查询字段都在索引里时，直接从索引返回结果，速度极快。

    # 索引的缺点

    • 占用额外磁盘空间:索引也是数据，表越大、索引越多，占用空间越大。
    • 增删改性能下降:插入、更新、删除时，MySQL 不仅要改数据，还要同步更新所有索引，导致写入变慢。
    • 增加查询优化器复杂度:索引过多时，MySQL 优化器需要花时间选择最优索引，可能选错索引，反而变慢。
    • 维护成本高:大表加索引、重建索引耗时耗 IO，业务高峰期不能操作。
    • 回表带来额外开销:普通非聚簇索引查询时，需要先查索引，再回表查数据，多一次 IO。
    • 索引并非万能:数据分布不均（如性别、状态只有两三个值）、小表、模糊查询 %xxx 等场景索引失效或没用。

    # 索引的使用场景

    • 对于非常小的表、大部分情况下简单的全表扫描比建立索引更高效。
    • 对于中到大型的表，索引就非常有效。
    • 但是对于特大型的表，建立和维护索引的代价将会随之增长。这种情况下，需要用到一种技术可以直接区分出需要查询的一组数据，而不是一条记录一条记录地匹配，例如可以使用分区技术。)
    量级 数据量范围 处理策略｜
    小表 ＜ 100 万行 不用索引、少量索引即可，全表扫描很快
    中表 100 万 ～ 1000 万行 建普通索引、联合索引，性能足够
    大表 1000 万 ～ 1 亿行 索引 + 分区表（按时间 / 范围分区）
    超大表 ＞ 1 亿行 分库分表（Sharding），单机扛不住

    # 什么是Mysql的分区表？

    MySQL 分区表 = 把一张大表，在物理上拆成多个小文件（分区），但逻辑上还是一张表。

      对应用层透明：SQL 不用改，还是查同一张表
  MySQL 内部自动把数据放到不同分区
  查询时只扫描符合条件的分区，跳过其他分区
  本质： 逻辑一张表，物理多块文件。
  MySQL 5.1 正式引入分区表功能、MySQL 5.5 / 5.6 / 5.7 / 8.0 持续增强、8.0 对分区性能、DDL 更友好

    分区表的原理

         1、按分区规则拆分数据
        根据你定义的规则（时间、HASH、LIST），把行数据分散到不同分区。
     2、分区裁剪（Partition Pruning）
        当查询带分区字段条件时，MySQL 直接跳过不相关分区，只扫需要的。
        这是分区表性能提升的核心原理。
     3、 存储结构
        每个分区对应独立的 .ibd 文件，各自管理索引和数据。
     所以：索引也被拆小了、扫描更快、清理历史数据极快（直接删分区）

    建表语法

      按 月份（month） 分区（常用）
  CREATE TABLE `order_info` (
  `id` bigint NOT NULL AUTO_INCREMENT,
  `order_no` varchar(64) NOT NULL,
  `user_id` bigint NOT NULL,
  `create_time` datetime NOT NULL,
  `amount` decimal(12,2) DEFAULT NULL,
  PRIMARY KEY (`id`, create_time)
  ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
  PARTITION BY RANGE (TO_DAYS(create_time)) (
  PARTITION p202501 VALUES LESS THAN (TO_DAYS('2025-02-01')),
  PARTITION p202502 VALUES LESS THAN (TO_DAYS('2025-03-01')),
  PARTITION p202503 VALUES LESS THAN (TO_DAYS('2025-04-01')),
  PARTITION p_future VALUES LESS THAN MAXVALUE
  );

    MySQL 支持 5 种标准分区:

    1. RANGE 范围（时间最常用）

        最常用：时间、数值区间
  sql：
     PARTITION BY RANGE (create_time)
  适合：订单时间、日志时间、金额区间。

    2. LIST 列表（固定值）

        给固定值分组
  比如按地区、状态、类型分区。
  sql:
  PARTITION BY LIST (city_id) (
     PARTITION p_north VALUES IN (11,12,13),
     PARTITION p_south VALUES IN (21,22,23)
  );
  适合：固定枚举值、地区、状态。

    3. HASH 哈希（均匀分散）

        MySQL 自动算哈希，均匀分散数据
  不用管值是什么，自动分到不同区，防数据倾斜。
  sql
     PARTITION BY HASH(user_id)
     PARTITIONS 8;  -- 分成8个区
  适合：用户 ID、订单 ID，均匀分布。

    4. KEY 自动哈希（简化版）

        比 HASH 更简单，不用自己写表达式
  sql
     PARTITION BY KEY(user_id)
     PARTITIONS 6;
  适合：不想写哈希算法，让 MySQL 自动分配。

    5. COLUMNS 多列分区（组合字段）

        可以用多个字段做分区条件
  sql:
     PARTITION BY RANGE COLUMNS(create_time, city_id) ...
  适合：复杂业务条件分区。
    项目 内容
    分区表是什么 逻辑上是一张表，物理上按规则拆成多个独立存储区域（分区文件），对应用层透明，SQL 无需修改
    优点 1. 查询更快：只扫描目标分区2. 删历史极快：直接 DROP 分区3. 索引更小更高效4. 对业务完全透明，无需改代码5. 分散 IO，缓解单表过大问题
    缺点 1. 必须带分区键查询才生效2. 主键 / 唯一索引必须包含分区键3. 不支持外键4. 分区过多会影响性能5. 无法突破单机 CPU/IO 瓶颈6. DDL 仍需操作全部分区
    适用场景 1. 日志、审计、监控类时序数据2. 订单、流水、账单等按时间查询3. 冷热数据分离，定期清理历史4. 单表过亿但仍在单机可承受范围
    不适用场景 1. 查询经常不带分区键2. 大量跨分区聚合、复杂 JOIN3. 超高并发写入，单机扛不住4. 需要跨机器水平扩容

    分区表 VS 分库分表 核心区别:

    对比维度 分区表 分库分表（Sharding）
    数据位置 同一数据库、同一 MySQL 实例 分布在多个库、多个实例 / 多台服务器
    对业务透明 完全透明，SQL 无需改造 不透明，需中间件或代码改造
    依赖组件 MySQL 原生支持，无需额外中间件 依赖 ShardingSphere、MyCat 等中间件
    解决问题 单表过大、查询慢、清理历史慢 单机性能瓶颈、高并发、海量存储
    扩容能力 单机内部分割，无法横向扩容 可无限水平扩容，支持集群
    使用复杂度 低，建表时指定分区规则即可 高，需处理分布式事务、跨库分页等
    事务支持 普通事务，支持完整 ACID 分布式事务复杂，成本高
    性能瓶颈 受限于单机 CPU、内存、IO 可通过加机器持续提升性能
    维护成本 低，管理同一张表 高，需管理多库多表、数据迁移、扩容
    典型使用阶段 中小规模、单表过大优化 高并发、大数据量、分布式架构
    一句话总结 单机内部优化手段 分布式水平扩展方案

    # MySQL 索引

    索引是在存储引擎层实现的，而不是在服务器层实现的，所以不同存储引擎具有不同的索引类型和实现。

    一、MySQL 索引有哪些？

    1. 按数据结构分

        B+ 树索引（最常用、默认）
  Hash 索引（Memory 引擎、InnoDB 自适应哈希）
  Full-Text 全文索引
  R-Tree 空间索引（GIS 地理数据）

    2. 按字段数量分

        普通单值索引
  联合索引（复合索引）

    3. 按唯一性约束分

        普通索引
  唯一索引
  主键索引（聚簇索引）

    4. 按物理存储分

        聚簇索引（Clustered Index）
  非聚簇索引（Secondary Index）

    5. 按业务用途分

           前缀索引
     覆盖索引
     唯一索引
     主键索引
     全文索引
     空间索引
    索引类型 英文 特点 优点 缺点 使用场景
    主键索引 Primary Key 唯一 + 非空，InnoDB 默认聚簇索引，叶子存整行数据 查找最快、无回表 只能有一个，更新代价高 每张表必须有，推荐自增 ID
    唯一索引 Unique 列值唯一，允许 NULL 约束重复值，查询快 插入 / 更新需校验唯一性 手机号、用户名、订单号
    普通索引 Index 无约束，仅加速查询 简单高效，开销小 无唯一性约束 常规查询字段，如状态、类型
    联合索引 Composite Index 多字段组合，遵循最左前缀原则 减少回表，优化多条件查询 字段顺序影响极大，不易维护 where a=? and b=? and c=?
    聚簇索引 Clustered 索引即数据，叶子节点存整行 范围查询极快，无需回表 插入无序会产生页分裂 InnoDB 主键天然是聚簇 ,B+ 树叶子节点 = 完整的一行数据
    非聚簇索引 Secondary 叶子存主键值，需回表查数据， 不影响数据存储，可建多个 查询需回表，多一次 IO 除主键外的所有索引（普通索引、唯一索引、联合索引等）都是非聚簇索引
    覆盖索引 Covering 查询所需字段全部在索引里 无需回表，性能极高 需精准设计索引 select id,name from user where id=?
    前缀索引 Prefix Index 只索引字符串前 N 个字符 节省空间，降低索引大小 无法 order by / 模糊优化 长字符串：varchar (255)、text
    全文索引 Fulltext 针对文本关键词搜索 比 like % xx% 快 N 倍 只支持文本，占用空间大 文章、商品描述搜索
    哈希索引 Hash Index 哈希表结构，等值查询极快 O (1) 查找，极快 不支持范围、排序、模糊 Memory 引擎、InnoDB 自适应哈希
    空间索引 R-Tree 地理坐标索引 高效地理位置查询 仅 GIS 场景使用 经距离查询、地图附近搜索

    # 1.聚簇索引

    聚簇索引并不是一种单独的索引类型，而是一种数据存储方式。InnoDB的表有且仅有一个聚簇索引。 聚簇索引简单理解就是将数据与索引放在一起，找到索引即找到了数据。换句话说，对于聚簇索引，其非叶子节点上存储的是索引字段的值，而叶子节点上存储的是对应记录的整行数据。

    - 定义: 决定了表中数据行的物理存储顺序。表数据本身就是聚簇索引的叶子节点。
- 如何创建：
    - 如果表定义了主键（PRIMARY KEY），InnoDB会使用主键作为聚簇索引。
    - 如果没有主键，InnoDB会选择第一个所有列都不为NULL的唯一索引作为聚簇索引。
    - 如果以上都没有，InnoDB会隐式创建一个名为GEN_CLUST_INDEX的ROWID作为聚簇索引。
    - 叶子节点内容： 存储了整个数据行的所有数据。

    # 2.非聚簇索引

    除了聚簇索引之外的所有索引，都称为非聚簇索引或二级索引。 非聚簇索引是指将索引与数据分开存储的一种方式。在非聚簇索引中，叶子节点包含索引字段的值以及指向数据页数据行的逻辑指针。

    - 定义： 叶子节点不包含行的全部数据，只存储了建立该索引的列的值和对应行的主键值。
- 如何创建： 我们通过 CREATE INDEX、UNIQUE 约束创建的，都是非聚簇索引。
- 叶子节点内容： 存储了索引列的值 + 对应记录的主键值。

    上图清晰地展示了：

        聚簇索引的叶子节点直接包含了完整的数据行。
    非聚簇索引的叶子节点只包含了索引列和主键值。
    通过非聚簇索引查询到数据后，如果需要获取未索引的列，必须通过主键ID回到聚簇索引中查找，这个过程就是回表(Bookmark Lookup)。
    # 深入理解“回表查询” (Bookmark Lookup)

    回表查询是理解InnoDB查询性能的核心关键。

    什么是回表？ 当SQL查询所需的数据列无法完全从非聚簇索引的叶子节点中获取时，InnoDB需要通过非聚簇索引记录的主键值，回到聚簇索引中重新定位并读取完整的行数据。这个过程就像查完目录后，还要根据页码翻到正文一样，称为回表。

    回表查询实战分析 假设我们有一张user 表，其结构如下：

    主键 (聚簇索引): id
有一个非聚簇索引: idx_name (name)
其他列: age, city

idINT PRIMARY KEY,

KEY idx_name(name)

    Case 1: 需要回表的查询

    SELECT * FROM user WHERE name = 'Alice';
    1

    或者

    SELECT id, name, age FROM user WHERE name = 'Alice'; -- age不在idx_name索引中
    1

    执行过程如下图所示:

       1. 在 idx_name 索引树中查找 name = 'Alice' 的记录。
   2. 在 idx_name 的叶子节点中，找到了 ('Alice', 18)（假设主键id=18）。
   3. 由于查询要获取age（或SELECT *），而idx_name索引中不包含age数据。
   4. 于是，使用找到的主键值 18，回到聚簇索引（主键索引）树中查找主键id=18的完整记录。
   5. 在聚簇索引的叶子节点中，拿到该行的所有数据 (18, 'Alice', 25, 'Beijing')。
   6. 返回结果。

    这个“从二级索引 -> 主键索引”的额外查询过程，就是回表。 它产生了额外的磁盘I/O，是性能的潜在瓶颈。

    Case 2: 避免回表的查询（索引覆盖）

    SELECT id, name FROM user WHERE name = 'Alice';
    1

    执行过程：

    1.在 idx_name 索引树中查找 name = 'Alice' 的记录。
2.在 idx_name 的叶子节点中，找到了 ('Alice', 18)。
3.发现要查询的 id 和 name 都包含在 idx_name 索引中（id是主键，必然存在）。
4.无需回表，直接将结果返回。

    这种情况下，EXPLAIN 的结果中会出现 Using index，代表本次查询使用了覆盖索引（Covering Index），是性能最高的查询方式。

    聚簇索引与非聚簇索引的区别

    特性 聚簇索引 非聚簇索引 (二级索引)
    数量 每表唯一 每表多个
    叶子节点内容 完整数据行 索引列 + 主键值
    依赖关系 独立 依赖聚簇索引（需要回表）
    查询效率 主键查询极快 可能需回表，效率较低

    回表查询是数据库工作的常态，但过多的回表就会成为性能瓶颈。 优化的核心思路就是：让查询尽可能在非聚簇索引的树上“结束战斗”，通过覆盖索引等技术，避免那最后一次昂贵的聚簇索引查找。

    # 面试题: 什么情况下会导致回表的出现？

    回表 = 二级索引查不到完整数据，需要再查主键索引 只要查询需要的字段不在二级索引里，就会回表。 避免回表的方法：建立覆盖索引。

    以下情况也不会出现回表:

    SELECT * FROM user WHERE id = 10;
    1

    聚簇索引叶子节点就是整行数据，不回表。

    效率排序: 聚簇索引（主键） > 覆盖索引（二级索引不回表） > 普通二级索引（回表） > 全表扫描

    # 3. 哈希索引

    哈希索引能以 O(1) 时间进行查找，但是失去了有序性，它具有以下限制:

    • 无法用于排序与分组；
    • 只支持精确查找，无法用于部分查找和范围查找。

    InnoDB 存储引擎有一个特殊的功能叫“自适应哈希索引”，当某个索引值被使用的非常频繁时，会在 B+Tree 索引之上再创建一个哈希索引，这样就让 B+Tree 索引具有哈希索引的一些优点，比如快速的哈希查找。

    # 4. 前缀索引

    对于 BLOB、TEXT 和 VARCHAR 类型的列，必须使用前缀索引，只索引开始的部分字符。

    对于前缀长度的选取需要根据索引选择性来确定。

    辅助索引的叶子节点的 data 域记录着主键的值，因此在使用辅助索引进行查找时，需要先查找到主键值，然后再到主索引中进行查找。

    # 5. 全文索引

    MyISAM 存储引擎支持全文索引，用于查找文本中的关键词，而不是直接比较是否相等。查找条件使用 MATCH AGAINST，而不是普通的 WHERE。

    全文索引一般使用倒排索引实现，它记录着关键词到其所在文档的映射。

    什么是倒排索引？

    把 “文档→关键词” 的正向关系，反过来变成 “关键词→文档” 的索引结构，是搜索引擎、数据库全文检索的核心原理。

      1. 正向索引（正常思路）
  每篇文档里有哪些词：
  文档 1：我 爱 北京 天安门
  文档 2：北京 欢迎 你
  文档 3：天安门 很 雄伟
  想查 “北京”， 就要遍历所有文档，效率很低。
  2. 倒排索引（反过来存）
     记录每个词出现在哪些文档里：
     北京：文档 1、文档 2
     天安门：文档 1、文档 3
     爱：文档 1
     欢迎：文档 2
     ……
     这样搜索 “北京” 时，直接查表就能拿到结果，速度极快。
     3. 实际结构更完整
        真实倒排索引一般包含：
        词（Term）
        出现的文档 ID 列表（Posting List）
        词频、位置、权重（用于排序）
        示例：
        北京 → [(文档1, 位置2), (文档2, 位置1)]
     4. 典型用途
        百度、Google 等搜索引擎
        Elasticsearch、Lucene 全文检索
        数据库 LIKE 优化、日志检索
     5. 为什么叫 “倒排”
        正向：文档 → 词
        倒排：词 → 文档
        关系反转，所以叫倒排索引。

    InnoDB 存储引擎在 MySQL 5.6.4 版本中也开始支持全文索引。

    # 6. 覆盖索引

    索引包含所有需要查询的字段的值。

    具有以下优点:

    • 索引通常远小于数据行的大小，只读取索引能大大减少数据访问量。
    • 一些存储引擎(例如 MyISAM)在内存中只缓存索引，而数据依赖于操作系统来缓存。因此，只访问索引可以不使用系统调用(通常比较费时)。

    # B+ Tree 原理

    B+ Tree 只有叶子节点存真实数据，非叶子节点只存索引键用于导航，所有叶子节点用双向链表串成有序链表。

    # 1. 数据结构

    它由三层组成：

    • 根节点（Root）
    • 内部节点（Internal / 非叶子节点）
    • 叶子节点（Leaf）

    B+ 树最核心的 4 个结构特点：

    • 只有叶子节点存数据，非叶子节点只做导航
    • 一个节点可以存大量 key，树非常矮
    • 所有叶子节点在同一层，高度平衡
    • 叶子节点有序 + 双向链表，范围查询极快

    B Tree 指的是 Balance Tree，也就是平衡树。平衡树是一颗查找树，并且所有叶子节点位于同一层。

    B+ Tree 是基于 B Tree 和叶子节点顺序访问指针进行实现，它具有 B Tree 的平衡性，并且通过顺序访问指针来提高区间查询的性能。

    在 B+ Tree 中，一个节点中的 key 从左到右非递减排列，如果某个指针的左右相邻 key 分别是 key[i] 和 key[i+1]， 且不为 null，则该指针指向节点的所有 key 大于等于 key[i] 且小于等于 key[i+1]。

    # 2. 操作

    进行查找操作时，首先在根节点进行二分查找，找到一个 key 所在的指针，然后递归地在指针所指向的节点进行查找。直到查找到叶子节点，然后在叶子节点上进行二分查找，找出 key 所对应的 data。

    插入删除操作记录会破坏平衡树的平衡性，因此在插入删除操作之后，需要对树进行一个分裂、合并、旋转等操作来维护平衡性。

    # 3. 与红黑树的比较

    红黑树等平衡树也可以用来实现索引，但是文件系统及数据库系统普遍采用 B+ Tree 作为索引结构，主要有以下两个原因:

    (一)更少的查找次数

    平衡树查找操作的时间复杂度等于树高 h，而树高大致为 O(h) = logdN (d 的h次方 等于 N？)，其中 d 为每个节点的出度。

    概念 英文 含义 通俗理解
    入度 In-degree 指向我的边（进来的） 有多少个「上级 / 前驱」指向我
    出度 Out-degree 我指向别人的边（出去的） 我有多少个「下级 / 后继」

    二者都是针对某一个节点计算的，是图论 / 数据结构的基础概念。

    树高度 = log分叉数总数据量

    红黑树的出度为 2，而 B+ Tree 的出度一般都非常大，所以红黑树的树高 h 很明显比 B+ Tree 大非常多，检索的次数也就更多。

    (二)利用计算机预读特性

    为了减少磁盘 I/O，磁盘往往不是严格按需读取，而是每次都会预读。预读过程中，磁盘进行顺序读取，顺序读取不需要进行磁盘寻道，并且只需要很短的旋转时间，因此速度会非常快。

    操作系统一般将内存和磁盘分割成固态大小的块，每一块称为一页，内存与磁盘以页为单位交换数据。数据库系统将索引的一个节点的大小设置为页的大小，使得一次 I/O 就能完全载入一个节点，并且可以利用预读特性，相邻的节点也能够被预先载入。

    # 1. B+Tree 索引

    是大多数 MySQL 存储引擎的默认索引类型。

    因为不再需要进行全表扫描，只需要对树进行搜索即可，因此查找速度快很多。除了用于查找，还可以用于排序和分组。

    可以指定多个列作为索引列，多个索引列共同组成键。

    适用于全键值、键值范围和键前缀查找，其中键前缀查找只适用于最左前缀查找。如果不是按照索引列的顺序进行查找，则无法使用索引。

    InnoDB 的 B+Tree 索引分为主索引和辅助索引。

    主索引的叶子节点 data 域记录着完整的数据记录，这种索引方式被称为聚簇索引。因为无法把数据行存放在两个不同的地方，所以一个表只能有一个聚簇索引。

    # 面试题: 为什么 MySQL InnoDB 索引用 B+ 树，而不用 B 树、红黑树？

    1. 为什么不用 红黑树？

       1、 红黑树是二叉树，数据量大时树高度极高
     每一层只有 2 个分叉
     千万级数据树高会达到 20~30 层
     每一层对应一次磁盘 I/O，I/O 次数太多，速度极慢
 2、 不适合范围查询
     红黑树叶子节点不连续
     范围查询需要频繁回溯、遍历，效率低
 3、 CPU 缓存局部性差
     节点分散，无法利用磁盘预读特性

    2. 为什么不用 B 树？

       1、 B 树的非叶子节点也存数据
     导致单个磁盘页（16KB）能存放的索引数量变少
     树高度变高，磁盘 I/O 变多
  范围查询效率低
 2、 B 树叶子节点不相连
     范围查询需要从根节点重新查找，无法顺序遍历
 3、 查询路径不稳定 
  可能在非叶子节点就命中数据,性能不稳定，不利于数据库稳定查询

    3. 为什么要用B+树

        1. 树极矮，磁盘 I/O 最少（最重要）
     非叶子节点只存 key，16KB 一页能存上千个 key
     千万级数据，树高通常 3~4 层
     查任意数据，最多 3~4 次磁盘 I/O，速度提升百倍
  2. 范围查询 / 排序 极速
        叶子节点是有序双向链表：
        WHERE id BETWEEN 100 AND 200
        直接找到起点，顺着链表遍历即可，无需回溯根节点
        这是数据库最核心的查询场景
  3. 查询效率 100% 稳定
        所有数据都在叶子节点，任何查询都必须走到叶子节点，性能稳定可预测。
  4. 全表扫描更快
        遍历全表只需遍历叶子节点链表，B 树必须遍历整棵树，效率差很多。

    基于以上sql，大概估算一下 四层高度的B+树可以存多少条数据

    B+树高度=4（Root -> Internal -> Internal -> Leaf）”来估算容量；因为 InnoDB 真实装填受很多细节影响（页内开销、记录头、实际字符串长度分布等），所以这是近似范围而不是精确上限。

    公式: N ≈ B^2 * L B代表Root页 与 Internal页 每一层 出度的数量。 L代表Leaf 每页能放多少行

    1. 统一假设（便于计算）

        innodb_page_size = 16KB，取 16KB = 16384B
  页头/目录/尾部等占用后，给估算用的“有效空间”取 ~14KB = 14336B
  聚簇索引：PRIMARY KEY (id)，因此 Leaf 页存整行（包含 id, order_no, user_id, create_time, amount）
  ROW_FORMAT=DYNAMIC 只影响变长列可能“页外存储”的细节，这里用记录开销做近似（你给的平均 order_no=63 通常不会极端导致记录爆炸变大）

    2. 估算 Leaf 单条 record 大小（从而算 Leaf 每页能放多少行 L）

        字段大致估算（utf8mb4）：
  
     1、order_no VARCHAR(64)
        平均字符数 63，utf8mb4 每字符约 4B  数据约：63 * 4 = 252B
        VARCHAR 长度前缀：通常取 ~2B（64*4=256B 上限附近一般需要 2 字节长度前缀的量级）
        合计：order_no ≈ 254B
        固定长度字段（在叶子记录里都要有）
  
     2、id BIGINT：8B
     3、user_id BIGINT：8B
     4、create_time DATETIME：8B
  小计：24B
     5、amount DECIMAL(12,2):DECIMAL 的存储是按分组压缩的，数量级通常在 5~7B；这里取 ~6B 作为估算中心值。
     6、记录额外开销（record header / NULL 位图 / 变长字段管理等）取一个近似常数：~40B（数量级估算）
  所以单条 record 估算：
  record_size ≈ 254 + 24 + 6 + 40 = 324B 再给一个合理范围（考虑开销浮动和对齐）：
  record_size ≈ 320B ~ 350B
  Leaf 每页可放行数：
  L ≈ 14336 / record_size
  当 record_size=320B：L≈44.8
  当 record_size=350B：L≈41.0
  因此：
  L 大约在 41 ~ 45 行/页（中心值可取 ~43）
    1. 估算 Internal 页 fanout（B，决定一层能分多少子树）

    Internal 节点条目要存：

    项目 在 InnoDB Internal（内部页）里对应的内容 大致字节（8.0 典型估算） 备注
    separator key B+Tree 分隔键：这里是主键列 id（BIGINT）作为 internal key 8B BIGINT 在索引 key 里用于估算时通常按 8 字节处理
    child 指针 指向子节点的 child 指针（子页/子槽所在的位置） ~4B 常用估算按 page number page_no 4 字节量级（用于容量估算）
    record header / 状态元数据 Internal 记录在页内组织所需的元数据/头部信息 ~6B ~ 12B 取决于 InnoDB 具体记录格式、实现细节；用于估算取范围
    slot / 条目目录相关 Page Directory（槽数组）中与条目对应的定位信息 ~2B 常用估算按每槽 uint16 offset 量级（只做数量级估算）
    内部条目有效大小（汇总） Internal 中“一条条目/记录”的平均有效成本 ~20B ~ 26B 近似值：8 + 4 + (6~12) + 2
         B ≈ 14336 / internal_entry_size
     B(min) ≈ 14336/26 ≈ 551
     B(max) ≈ 14336/20 ≈ 716
     因此： B 大约在 550 ~ 720
    1. 总行数：N ≈ B^2 * L
    情况 B(扇出) L(每页行数) N ≈ B² * L
    保守 550 41 550² * 41 = 12,4M
    典型 650 43 650² * 43 = 18.2M
    偏乐观 720 45 720² * 45 = 23.3M

    结论 :在以上的条件下，并且确实“B+树高度=4”，那么大概容量是： 约 1200 万 ~ 2300 万 行（典型在 ~1800 万 附近）
