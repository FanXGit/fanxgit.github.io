MySQL - 性能优化

# MySQL - 性能优化

=============================

# 使用 Explain 进行分析

EXPLAIN 是 MySQL 分析 SELECT 查询性能的核心工具，执行后会返回查询的执行计划，帮你快速定位慢查询、索引失效、全表扫描等问题。

下面是最关键、必须掌握的字段，按重要性排序：

# 1、id（查询执行顺序）

作用：标识查询中操作表的执行顺序，决定先查哪张表。

  • id 相同：从上到下顺序执行
  • id 不同：数字越大越先执行（子查询优先）
  • id 为 NULL：最后执行（结果集合并 / 排序）

优化点：子查询嵌套过多会导致 id 层级复杂，优先改写成 JOIN 优化。

# 2、select_type（查询类型）

作用：判查询是简单查询还是复杂查询，定位查询复杂度。

核心值 含义
SIMPLE 简单查询（无子查询 / UNION），最优
PRIMARY 主查询（最外层查询）
SUBQUERY 子查询（性能较差）
DERIVED 派生表（临时表，性能一般）
UNION UNION 中的第二个及以后的查询

优化点：尽量避免 SUBQUERY/DERIVED，改用 JOIN 提升性能。

# 3、table（操作的表）

作用：显示当前行数据对应哪张表，多表联查时清晰区分。

可能是表名、别名、临时表名。

# 4. type（最重要：访问类型）

✅ 查询性能的核心指标，决定 MySQL 如何查找数据，性能从优到差：

system
> const > eq_ref > ref > range > index > ALL
1
2

关键值解释：

  • const：通过主键 / 唯一索引精准匹配 1 行（最快，最优）
  • eq_ref：JOIN 关联时，使用主键 / 唯一索引匹配（多表最优）
  • ref：普通索引匹配，返回多行（良好）
  • range：索引范围查询（> < between in）
  • index：遍历索引树（比全表扫描快）
  • ALL：全表扫描（最差！必须优化）

优化标准：业务查询至少达到 range 级别，优先 ref/const，杜绝 ALL。

# 5. possible_keys（可能用到的索引）

作用：MySQL 理论上可能选择的索引，仅作参考。

为 NULL：无可用索引，大概率全表扫描。

# 6. key（实际用到的索引）

✅ 判断索引是否生效的核心字段

  • 显示 MySQL 真正使用的索引
  • 为 NULL：索引失效（最需要优化）
  • 多表查询：显示关联使用的索引

优化点：预期索引未出现在 key 中，说明索引创建 / 使用错误。

# 7. key_len（使用的索引长度）

作用：判断索引使用程度（是否用了联合索引的全部字段）

  • 数值越小：索引使用越精准
  • 联合索引：长度越大，说明命中的索引字段越多
  • 例子：联合索引 idx(a,b,c)，key_len 只包含 a → 只命中第一个字段。

# 8. ref（索引匹配的字段 / 常量）

作用：显示与索引比较的值

  • const：用常量匹配索引（最优）
  • 表字段：多表 JOIN 时关联的字段

# 9. rows（预估扫描行数）

作用：MySQL 预估需要扫描的数据行数

  • 数值越小：查询效率越高
  • 全表扫描时，rows 接近表总数据量 优化点：通过索引把 rows 从几十万降到个位数 / 十位数。

# 10. Extra（额外重要信息）

作用：补充查询细节，包含大量优化预警

  • 必须警惕的关键字：
  • Using filesort：文件排序（严重！MySQL 无法用索引排序，需额外排序）
  • Using temporary：临时表（严重！查询需创建临时表存储结果）
  • Using index：覆盖索引（最优！直接从索引取数据，不回表）
  • Using where：用 WHERE 条件过滤数据（正常）

优化点：必须消除 Using filesort 和 Using temporary。

** 快速化口诀**

  • 看 type：杜绝 ALL，优先 ref/const
  • 看 key：必须有索引，不能为 NULL
  • 看 rows：扫描行数越小越好
  • 看 Extra：消灭 filesort/temporary

总结

  • type 是性能等级，ALL 全表扫描必须优化；
  • key 是索引是否生效的直接证明；
  • Extra 中的 Using filesort/temporary 是慢查询重灾区；
  • 核心目标：让查询走索引、少扫描、不额外排序。

# 索引优化策略

# 1. 索引类型选择

  • 主键索引：表必须有主键，建议使用自增整数类型
  • 唯一索引：确保列值唯一，适合邮箱、手机号等字段
  • 普通索引：加速查询，适合频繁查询的字段
  • 联合索引：遵循最左前缀原则，适合多字段查询
  • 全文索引：适合文本搜索，MySQL 5.6+ 支持

# 2. 索引设计原则

  • 选择性高：索引列的区分度越高越好
  • 长度适中：索引列长度不宜过长，影响性能
  • 最左前缀：联合索引按顺序使用，避免跳过中间列
  • 覆盖索引：包含查询所需的所有字段，避免回表
  • 避免冗余：删除不必要的重复索引

# 3. 索引失效场景

  • 模糊查询：LIKE '%xxx' 会导致索引失效
  • 函数操作：WHERE DATE(create_time) = '2023-01-01'
  • 类型转换：字符串与数字比较
  • OR 条件：多个条件使用 OR 且不全有索引
  • NOT IN / NOT EXISTS：可能导致全表扫描
  • 联合索引：违反最左前缀原则

# 数据库结构优化

# 1. 表结构设计

  • 范式化：遵循第三范式，减少数据冗余
  • 反范式化：适当冗余以提高查询性能
  • 数据类型选择：
    • 优先使用整数类型（TINYINT, INT, BIGINT）
    • 字符串使用 VARCHAR 而非 CHAR（变长更节省空间）
    • 时间类型使用 DATETIME 或 TIMESTAMP
    • 避免使用 TEXT/BLOB 类型，影响性能

# 2. 分区表

  • 水平分区：按行分区，适合大表
  • 垂直分区：按列分区，适合宽表
  • 分区类型：
    • RANGE 分区：按范围（如日期）
    • LIST 分区：按列表值
    • HASH 分区：按哈希值
    • KEY 分区：按主键或唯一键

# 3. 分库分表

  • 水平分表：将数据分散到多个表
  • 垂直分库：按业务模块分库
  • 分库分表策略：
    • 范围分片
    • 哈希分片
    • 一致性哈希

# 优化数据访问

# 1. 减少请求的数据量

  • 只返回必要的列: 最好不要使用 SELECT * 语句。
  • 只返回必要的行: 使用 LIMIT 语句来限制返回的数据。
  • 缓存重复查询的数据: 使用缓存可以避免在数据库中进行查询，特别在要查询的数据经常被重复查询时，缓存带来的查询性能提升将会是非常明显的。

# 2. 减少服务器端扫描的行数

最有效的方式是使用索引来覆盖查询。

# 重构查询方式

# 1. 切分大查询

一个大查询如果一次性执行的话，可能一次锁住很多数据、占满整个事务日志、耗尽系统资源、阻塞很多小的但重要的查询。

-- 不推荐
DELETE FROM messages WHERE create_time < DATE_SUB(NOW(), INTERVAL 3 MONTH);

-- 推荐
SET @rows_affected = 1;
WHILE @rows_affected > 0 DO
    DELETE FROM messages 
    WHERE create_time < DATE_SUB(NOW(), INTERVAL 3 MONTH) 
    LIMIT 10000;
    SET @rows_affected = ROW_COUNT();
    DO SLEEP(0.1); -- 适当休息，避免系统负载过高
END WHILE;

# 2. 分解大连接查询

将一个大连接查询分解成对每一个表进行一次单表查询，然后将结果在应用程序中进行关联，这样做的好处有:

  • 让缓存更高效。对于连接查询，如果其中一个表发生变化，那么整个查询缓存就无法使用。而分解后的多个查询，即使其中一个表发生变化，对其它表的查询缓存依然可以使用。
  • 分解成多个单表查询，这些单表查询的缓存结果更可能被其它查询使用到，从而减少冗余记录的查询。
  • 减少锁竞争；
  • 在应用层进行连接，可以更容易对数据库进行拆分，从而更容易做到高性能和可伸缩。
  • 查询本身效率也可能会有所提升。例如下面的例子中，使用 IN() 代替连接查询，可以让 MySQL 按照 ID 顺序进行查询，这可能比随机的连接要更高效。
-- 不推荐
SELECT * FROM tag
JOIN tag_post ON tag_post.tag_id = tag.id
JOIN post ON tag_post.post_id = post.id
WHERE tag.tag = 'mysql';

-- 推荐
SELECT * FROM tag WHERE tag = 'mysql'; -- 假设返回 tag_id = 1234
SELECT * FROM tag_post WHERE tag_id = 1234;
SELECT * FROM post WHERE post.id IN (123, 456, 567, 9098, 8904);

# SQL 语句优化技巧

# 1. WHERE 子句优化

  • 避免 NULL 判断：使用默认值代替 NULL
  • 使用 BETWEEN 代替 IN：BETWEEN 1 AND 100 比 IN (1,2,...,100) 更高效
  • 使用 EXISTS 代替 IN：EXISTS (SELECT 1 FROM table WHERE ...) 比 IN (SELECT ... FROM table) 更高效
  • 避免使用 != 或 <>：会导致全表扫描
  • 合理使用 OR：确保 OR 两边的字段都有索引

# 2. JOIN 优化

  • 小表驱动大表：SELECT * FROM small_table JOIN big_table ON ...
  • 使用 INNER JOIN：比 LEFT/RIGHT JOIN 更高效
  • 避免笛卡尔积：确保 JOIN 条件正确
  • 使用 USING 或 ON：明确连接条件

# 3. 排序优化

  • 使用索引排序：避免 Using filesort
  • ** LIMIT 优化**：LIMIT 10000, 10 会扫描 10010 行，可优化为 WHERE id > 10000 LIMIT 10
  • 避免在 ORDER BY 中使用函数

# 服务器配置优化

# 1. 内存配置

  • innodb_buffer_pool_size：建议设置为服务器内存的 70-80%
  • key_buffer_size：MyISAM 表索引缓存，建议 16-32M
  • query_cache_size：查询缓存，MySQL 8.0 已移除
  • sort_buffer_size：排序缓存，建议 2-4M
  • read_buffer_size：顺序读取缓存，建议 1-2M

# 2. 并发配置

  • max_connections：最大连接数，根据服务器性能调整
  • innodb_thread_concurrency：InnoDB 并发线程数
  • wait_timeout：连接超时时间，避免连接泄漏

# 3. 日志配置

  • innodb_log_file_size：InnoDB 日志文件大小，建议 256M-1G
  • innodb_log_buffer_size：日志缓冲区大小，建议 16-64M
  • slow_query_log：开启慢查询日志
  • long_query_time：慢查询阈值，建议 1-2 秒

# 性能监控与分析

# 1. 慢查询日志

  • 开启慢查询日志：slow_query_log = 1
  • 设置阈值：long_query_time = 1
  • 分析工具：
    • mysqldumpslow：分析慢查询日志
    • pt-query-digest：更强大的分析工具

# 2. 监控工具

  • SHOW PROCESSLIST：查看当前运行的进程
  • SHOW GLOBAL STATUS：查看全局状态
  • SHOW ENGINE INNODB STATUS：查看 InnoDB 状态
  • Performance Schema：详细的性能数据
  • 第三方工具：
    • MySQL Enterprise Monitor
    • Percona Monitoring and Management (PMM)
    • Navicat Monitor

# 3. 常用诊断命令

-- 查看当前连接数
SHOW STATUS LIKE 'Threads%';

-- 查看慢查询数量
SHOW GLOBAL STATUS LIKE '%slow%';

-- 查看索引使用情况
SHOW GLOBAL STATUS LIKE '%index%';

-- 查看表状态
SHOW TABLE STATUS LIKE 'table_name';

-- 分析表（更新统计信息）
ANALYZE TABLE table_name;

-- 优化表（整理碎片）
OPTIMIZE TABLE table_name;

# 缓存策略

# 1. 查询缓存

  • 注意：MySQL 8.0 已移除查询缓存
  • 适用场景：只读或极少更新的表
  • 配置：query_cache_type = ON, query_cache_size = 64M

# 2. 应用层缓存

  • Redis：高性能内存数据库，适合缓存热点数据
  • Memcached：简单高效的内存缓存
  • 本地缓存：如 Guava Cache、Caffeine 等

# 3. 缓存策略

  • 缓存穿透：使用布隆过滤器
  • 缓存雪崩：设置随机过期时间
  • 缓存击穿：使用互斥锁或预热

# 事务与锁优化

# 1. 事务隔离级别

  • READ UNCOMMITTED：最低级别，可能脏读
  • READ COMMITTED：避免脏读
  • REPEATABLE READ：MySQL 默认，避免脏读和不可重复读
  • SERIALIZABLE：最高级别，完全串行化

# 2. 锁优化

  • 减少锁持有时间：事务尽可能短
  • 使用行级锁：避免表级锁
  • 合理使用索引：索引可以减少锁范围
  • 避免死锁：按相同顺序访问资源
  • 使用乐观锁：适合并发冲突少的场景

# 总结

=============================

MySQL 性能优化是一个系统工程，需要从多个维度入手：

  1. 索引优化：创建合适的索引，避免索引失效
  2. SQL 优化：编写高效的 SQL 语句，使用 EXPLAIN 分析
  3. 表结构优化：合理设计表结构，使用合适的数据类型
  4. 服务器配置：根据硬件资源调整配置参数
  5. 缓存策略：使用多级缓存提升性能
  6. 监控分析：持续监控性能，及时发现问题

优化的核心原则是：减少数据访问量，提高查询效率，合理利用系统资源。通过不断的监控、分析和调整，才能使 MySQL 数据库达到最佳性能状态。
