# 前言

天天和数据库打交道，一天能写上几十条 SQL 语句，但你知道我们的系统是如何和数据库交互的吗？MySQL 如何帮我们存储数据、又是如何帮我们管理事务？....是不是感觉真的除了写几个 「select * from dual」外基本脑子一片空白？这篇文章就将带你走进 MySQL 的世界，让你彻底了解系统到底是如何和 MySQL 交互的，MySQL 在接受到我们发送的 SQL 语句时又分别做了哪些事情。

本文将从以下几个方面详细介绍 SQL 语句在 MySQL 中的执行过程：

  • 客户端与 MySQL 的连接建立
  • MySQL 服务器端的处理流程
  • 存储引擎层的工作机制
  • 事务日志的作用
  • SQL 执行的完整流程
  • 最佳实践与性能优化

# 客户端与 MySQL 的连接

# MySQL 驱动

我们的系统在和 MySQL 数据库进行通信时，需要通过 MySQL 驱动来建立连接。MySQL 驱动负责底层的通信协议实现，帮助我们与数据库建立 TCP 连接，发送 SQL 语句并接收执行结果。

# 数据库连接池

如果每次 SQL 请求都创建一个新的连接，使用完后再关闭，会带来巨大的性能开销：

  • 连接建立：TCP 三次握手
  • 连接关闭：TCP 四次挥手
  • 资源消耗：每个连接都需要占用系统资源

ps: 什么是TCP三次握手？ 什么是TCP四次挥手？

三次握手的过程；
    客户端 → 服务器
        “我想连你，你在吗？”（SYN）
    服务器 → 客户端
        “我在，你能收到我吗？”（SYN+ACK）
    客户端 → 服务器
        “我收到了，连接正式建立！”（ACK）

四次挥手:
    客户端 → 服务器
        “我发完数据了，要关了。”（FIN）
    服务器 → 客户端
        “收到，我知道了。但我可能还有数据没发完。”（ACK）
    服务器 → 客户端
        “我也发完了，可以关了。”（FIN）
    客户端 → 服务器
        “收到，正式关闭。”（ACK）

数据库连接池 解决了这个问题：

  • 维护一定数量的数据库连接
  • 应用程序从池中获取连接，使用后归还
  • 避免了频繁创建和销毁连接的开销

常见的数据库连接池：

  • Druid：阿里巴巴开源，功能丰富，监控强大
  • HikariCP：Spring Boot 默认，性能优异
  • C3P0：老牌连接池，稳定可靠
  • DBCP：Apache 提供，使用广泛
连接池 性能 功能 监控 维护状态 推荐场景
HikariCP ⭐⭐⭐⭐⭐ 最强 ⭐⭐⭐ ⭐⭐ 活跃 微服务、高并发
Druid ⭐⭐⭐⭐ ⭐⭐⭐⭐⭐ 最全 ⭐⭐⭐⭐⭐ 最强 活跃 企业项目、需要监控
C3P0 ⭐⭐ ⭐⭐⭐ ⭐ 停止更新 老旧系统
DBCP2 ⭐⭐⭐ ⭐⭐⭐ ⭐⭐ 稳定维护 简单小项目

# MySQL 连接处理

MySQL 服务器端也有连接管理机制：

  • 连接池：管理客户端连接
  • 线程处理：每个连接由独立的线程处理
  • 连接限制：通过 max_connections 参数控制最大连接数

当客户端发送 SQL 请求时，MySQL 服务器会：

  1. 接收连接请求
  2. 分配线程处理
  3. 执行 SQL 语句
  4. 返回执行结果
  5. 保持连接或关闭连接

# MySQL 服务器端处理流程

当 MySQL 服务器接收到客户端的 SQL 请求后，会经过以下几个步骤的处理：

# SQL 接口

SQL 接口 是 MySQL 服务器与客户端交互的入口，负责接收客户端发送的 SQL 语句，并将执行结果返回给客户端。

# 查询解析器

查询解析器 将 SQL 语句解析成 MySQL 内部可理解的语法树：

  1. 词法分析：识别 SQL 语句中的关键字、表名、列名等
  2. 语法分析：检查 SQL 语句的语法是否正确
  3. 语义分析：检查表和列是否存在，用户是否有操作权限

# 查询优化器

查询优化器 负责选择最优的执行计划：

  1. 成本估算：计算不同执行计划的成本

    • IO 成本：从磁盘读取数据的成本（默认读取一页成本为 1）
    • CPU 成本：数据处理和计算的成本（默认检测一条记录成本为 0.2）

  2. 执行计划选择：选择成本最低的执行计划

    • 选择合适的索引
    • 确定表的连接顺序
    • 选择连接方式（Nested Loop、Hash Join 等）

# 执行器

执行器 根据优化器生成的执行计划，调用存储引擎的接口执行 SQL 语句：

  1. 调用存储引擎接口获取数据
  2. 处理结果集
  3. 返回执行结果给客户端

# 存储引擎层

存储引擎是 MySQL 中负责数据存储和检索的核心组件，InnoDB 是 MySQL 5.5 以后的默认存储引擎。我们以 InnoDB 为例，介绍存储引擎层的工作机制：

# Buffer Pool

Buffer Pool 是 InnoDB 最重要的内存结构，用于缓存数据页和索引页：

  • 作用：减少磁盘 I/O，提高查询性能
  • 大小：通过 innodb_buffer_pool_size 参数设置，建议为服务器内存的 70-80%
  • 管理：使用 LRU 算法管理缓存页

工作流程：

  1. 当需要访问数据时，先在 Buffer Pool 中查找
  2. 如果找到（缓存命中），直接使用
  3. 如果没找到（缓存未命中），从磁盘加载到 Buffer Pool 中

# 事务日志

InnoDB 使用多种日志来保证事务的 ACID 特性：

# Undo Log

Undo Log 记录数据被修改前的状态，用于：

  • 事务回滚：当事务失败时，恢复到修改前的状态
  • MVCC：多版本并发控制，为快照读提供历史版本

工作流程：

  1. 在修改数据前，将原始数据写入 Undo Log
  2. 如果事务失败，根据 Undo Log 回滚数据
  3. 事务提交后，Undo Log 可能会被 purge 线程清理

# Redo Log

Redo Log 记录数据被修改后的状态，用于：

  • 崩溃恢复：当数据库宕机时，恢复未持久化的数据
  • 数据持久化：确保事务的持久性

特点：

  • 物理日志：记录数据页的修改
  • 循环写入：固定大小，循环覆盖
  • 先写日志：WAL (Write-Ahead Logging) 机制

刷盘策略：通过 innodb_flush_log_at_trx_commit 参数控制

  • 0：每秒刷盘，可能丢失最多 1 秒的数据
  • 1：每次事务提交刷盘，最安全
  • 2：先刷到 OS cache，每秒刷盘

# Bin Log

Bin Log 是 MySQL 级别的日志，用于：

  • 主从复制：将数据变更复制到从库
  • 数据恢复：通过 bin log 恢复数据

特点：

  • 逻辑日志：记录 SQL 语句或行变更
  • 追加写入：文件大小达到阈值后创建新文件
  • 所有引擎通用：不限于 InnoDB

格式：

  • STATEMENT：记录 SQL 语句，体积小但可能导致主从不一致
  • ROW：记录行变更，安全但体积大
  • MIXED：混合模式，自动选择合适的格式

刷盘策略：通过 sync_binlog 参数控制

  • 0：依赖操作系统刷新，可能丢失数据
  • 1：每次事务提交刷盘，最安全

两阶段提交：

  1. 准备阶段：写入 redo log，标记为 prepare 状态
  2. 提交阶段：写入 bin log，然后将 redo log 标记为 commit 状态

这样确保了 redo log 和 bin log 的一致性，避免主从复制数据不一致。

# SQL 执行完整流程

弄的pwd老师唱的 现在我们来总结一下一条 SQL 语句的完整执行流程：

# 更新操作流程

以 UPDATE students SET stuName = '小强' WHERE id = 1 为例：

  1. 客户端连接：通过 MySQL 驱动从连接池获取连接
  2. SQL 发送：将 SQL 语句发送到 MySQL 服务器
  3. 连接处理：MySQL 服务器分配线程处理请求
  4. SQL 解析：解析器将 SQL 解析成语法树
  5. 查询优化：优化器生成最优执行计划
  6. 执行器执行：调用存储引擎接口执行 SQL
  7. Buffer Pool 操作：
    • 在 Buffer Pool 中查找数据，未找到则从磁盘加载
    • 对记录加独占锁
    • 将原始数据写入 Undo Log
    • 在 Buffer Pool 中执行更新操作
    • 将更新后的数据写入 Redo Log Buffer
  8. 事务提交：
    • 将 Redo Log Buffer 刷入 Redo Log 文件
    • 写入 Bin Log 文件
    • 在 Redo Log 中添加 commit 标记
  9. 后台刷盘：后台线程将 Buffer Pool 中的脏数据刷入磁盘
  10. 结果返回：将执行结果返回给客户端
  11. 连接归还：客户端将连接归还到连接池

# 读取操作流程

以 SELECT stuName,age,sex FROM students WHERE id=1 为例：

  1. 客户端连接：通过 MySQL 驱动从连接池获取连接
  2. SQL 发送：将 SQL 语句发送到 MySQL 服务器
  3. 连接处理：MySQL 服务器分配线程处理请求
  4. SQL 解析：解析器将 SQL 解析成语法树
  5. 查询优化：优化器生成最优执行计划
  6. 执行器执行：调用存储引擎接口执行 SQL
  7. Buffer Pool 操作：
    • 在 Buffer Pool 中查找数据，未找到则从磁盘加载
    • 如果是快照读，使用 MVCC 机制获取历史版本
    • 如果是当前读，对记录加锁
  8. 结果返回：将查询结果返回给客户端
  9. 连接归还：客户端将连接归还到连接池

# 最佳实践与性能优化

# 连接管理

  • 使用连接池：配置合理的连接池大小
  • 控制连接数：避免超过 max_connections 限制
  • 及时释放连接：使用完后及时归还到连接池

# SQL 优化

  • 使用索引：为查询条件和排序字段创建索引
  • 避免全表扫描：优化 WHERE 条件，使用覆盖索引
  • 合理设计表结构：避免过多字段，使用合适的数据类型
  • 优化 JOIN 查询：小表驱动大表，使用适当的 JOIN 类型

# 存储引擎优化

  • 配置 Buffer Pool：设置合适的 innodb_buffer_pool_size
  • 日志配置：
    • innodb_flush_log_at_trx_commit = 1（生产环境）
    • sync_binlog = 1（生产环境）
  • 磁盘配置：使用 SSD 存储，配置合理的 RAID 级别

# 监控与维护

  • 监控慢查询：开启慢查询日志，分析并优化慢查询
  • 定期优化表：使用 OPTIMIZE TABLE 整理碎片
  • 监控系统状态：关注 CPU、内存、I/O 等指标
  • 备份策略：定期备份数据，确保数据安全

# 总结

本文详细介绍了 MySQL 中一条 SQL 语句的执行过程，从客户端连接到服务器端处理，再到存储引擎层的工作机制，最后到事务提交和数据持久化。

核心要点：

  • 连接管理：通过连接池提高连接效率
  • 服务器处理：解析、优化、执行 SQL 语句
  • 存储引擎：Buffer Pool 缓存数据，提高性能
  • 事务日志：Undo Log 用于回滚，Redo Log 用于崩溃恢复
  • Bin Log：用于主从复制和数据恢复
  • 两阶段提交：确保 Redo Log 和 Bin Log 的一致性

通过了解 SQL 的执行过程，我们可以更好地理解 MySQL 的工作原理，从而写出更高效的 SQL 语句，配置更合理的数据库参数，提高系统的整体性能。

希望本文能帮助你对 MySQL 的内部工作机制有更深入的了解，为你的数据库优化工作提供参考。
