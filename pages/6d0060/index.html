# 为什么需要分表分库

当数据库达到一定规模时，会面临以下挑战：

  • 性能瓶颈：单表数据量过大（如超过 1000 万行），查询速度显著下降
  • 存储限制：单库存储容量有限，难以满足业务增长需求
  • 并发压力：高并发场景下，单库处理能力不足
  • 维护困难：大表备份、恢复时间长，影响业务可用性

分表分库是解决这些问题的有效方案，通过将数据分散到多个数据库或表中，提高系统的扩展性和性能。

# 水平切分

水平切分又称为 Sharding，它是将同一个表中的记录拆分到多个结构相同的表中。

当一个表的数据不断增多时，Sharding 是必然的选择，它可以将数据分布到集群的不同节点上，从而缓解单个数据库的压力。

# 水平切分的适用场景

  • 数据量大：单表数据量超过 1000 万行
  • 查询频繁：需要高频查询的大表
  • 数据增长快：业务数据增长迅速，预计会持续增长

# 水平切分实例

# 示例 1：用户表水平切分

假设用户表 user 有 5000 万行数据，按用户 ID 哈希分表：

-- 原始表
CREATE TABLE `user` (
  `id` BIGINT PRIMARY KEY,
  `name` VARCHAR(50),
  `email` VARCHAR(100),
  `phone` VARCHAR(20),
  `create_time` DATETIME
);

-- 分表后（分 4 个表）
CREATE TABLE `user_0` LIKE `user`;
CREATE TABLE `user_1` LIKE `user`;
CREATE TABLE `user_2` LIKE `user`;
CREATE TABLE `user_3` LIKE `user`;

-- 数据路由规则
-- user_id % 4 得到分表索引

# 垂直切分

垂直切分是将一张表按列切分成多个表，通常是按照列的关系密集程度进行切分，也可以利用垂直切分将经常被使用的列和不经常被使用的列切分到不同的表中。

在数据库的层面使用垂直切分将按数据库中表的密集程度部署到不同的库中，例如将原来的电商数据库垂直切分成商品数据库、用户数据库等。

# 垂直切分的适用场景

  • 表结构复杂：表字段过多（如超过 50 个字段）
  • 访问模式差异大：部分字段访问频繁，部分字段访问较少
  • 数据类型差异：包含大字段（如 TEXT、BLOB）影响查询性能

# 垂直切分实例

# 示例 1：用户表垂直切分

-- 原始表
CREATE TABLE `user` (
  `id` BIGINT PRIMARY KEY,
  `name` VARCHAR(50),
  `email` VARCHAR(100),
  `phone` VARCHAR(20),
  `address` TEXT, -- 大字段
  `bio` TEXT,     -- 大字段
  `create_time` DATETIME,
  `update_time` DATETIME
);

-- 垂直切分后
-- 核心信息表（频繁访问）
CREATE TABLE `user_core` (
  `id` BIGINT PRIMARY KEY,
  `name` VARCHAR(50),
  `email` VARCHAR(100),
  `phone` VARCHAR(20),
  `create_time` DATETIME,
  `update_time` DATETIME
);

-- 扩展信息表（较少访问）
CREATE TABLE `user_extra` (
  `user_id` BIGINT PRIMARY KEY,
  `address` TEXT,
  `bio` TEXT
);

# Sharding 策略详解

# 1. 哈希取模策略

# 实现原理

-- 分片索引 = hash(key) % 分片数量
-- 例如：用户 ID 取模分 4 个库
shard_index = user_id % 4
1
2
3

# 优点

  • 数据分布均匀：理论上可以均匀分布数据
  • 实现简单：计算逻辑简单直观

# 缺点

  • 扩容困难：增加分片数量需要重新计算所有数据的分片位置
  • 热点问题：如果哈希键分布不均匀，可能导致某些分片数据过多

# 实例

// Java 代码实现
public int getShardIndex(long userId, int shardCount) {
    return (int) (userId % shardCount);
}

// 使用示例
int shardIndex = getShardIndex(10001L, 4); // 返回 1，使用 user_1 表

# 4. 一致性哈希策略

# 实现原理

  • 将数据和分片节点都映射到一个哈希环上
  • 数据按顺时针方向找到最近的分片节点

# 优点

  • 扩容影响小：只影响相邻的分片节点
  • 负载均衡：可以根据节点性能调整权重

# 缺点

  • 实现复杂：需要维护哈希环结构
  • 查询效率：需要计算哈希值并查找节点

# Sharding 存在的问题及解决方案

# 1. 事务问题

# 问题

分布式环境下，跨分片的操作无法保证事务的原子性。

# 解决方案

  1. 分布式事务：使用 XA 接口或两阶段提交（2PC）

    • 优点：保证事务一致性
    • 缺点：性能开销大，可能导致长时间锁

  2. 最终一致性：采用异步消息队列

    • 优点：性能高
    • 缺点：存在短暂的数据不一致

  3. 本地事务：将相关操作尽量放在同一个分片内

    • 优点：性能好，实现简单
    • 缺点：设计复杂度高

# 实例

// 使用 Seata 分布式事务框架
@GlobalTransactional
public void createOrderAndDeductInventory(OrderDTO order) {
    // 创建订单（可能在 order_db_1）
    orderService.create(order);
    // 扣减库存（可能在 product_db_2）
    inventoryService.deduct(order.getProductId(), order.getQuantity());
}

# 2. 跨分片查询问题

# 问题

原来的 JOIN 查询在分库分表后无法直接执行。

# 解决方案

  1. 应用层 JOIN：将查询分解为多个单表查询，在应用程序中进行 JOIN

    • 实例：
      // 1. 查询用户信息
User user = userService.getById(userId);
// 2. 查询用户订单
List<Order> orders = orderService.getByUserId(userId);
// 3. 应用层关联
user.setOrders(orders);

  2. 全局表：将公共数据（如字典表）在每个分片上都存储一份

    • 优点：避免跨分片查询
    • 缺点：需要同步维护多个副本

  3. ER 分片：将相关数据放在同一个分片内

    • 实例：用户和用户订单使用相同的分片策略

# 3. ID 唯一性问题

# 问题

分库分表后，自增 ID 无法保证全局唯一。

# 解决方案

  1. 全局唯一 ID：使用 GUID/UUID

    • 优点：实现简单
    • 缺点：字符串长度长，索引性能差

  2. 分段 ID：为每个分片指定 ID 范围

    • 实例：

# 4. 跨分片排序和分页问题

# 问题

分库分表后，ORDER BY 和 LIMIT 操作变得复杂。

# 解决方案

  1. 全局排序：在应用层收集所有分片的数据后进行排序

    • 优点：实现简单
    • 缺点：数据量大时性能差

  2. 分片内排序 + 应用层合并：

    • 实例：
      // 1. 每个分片内排序并获取前 N 条
List<List<Order>> shardResults = new ArrayList<>();
for (int i = 0; i < shardCount; i++) {
    shardResults.add(orderService.getTopNByShard(i, 100));
}
// 2. 应用层合并排序
List<Order> finalResults = mergeSort(shardResults).subList(0, 10);

# 5. 数据迁移和扩容问题

# 问题

分库分表后，数据迁移和扩容变得复杂。

# 解决方案

  1. 双写迁移：

    • 步骤：
      1. 同时向旧分片和新分片写入数据
      2. 同步历史数据
      3. 切换读写到新分片
      4. 停止旧分片写入

  2. 使用工具：

    • Canal：监听 MySQL binlog 进行数据同步
    • MyBatis-Plus Sharding：提供数据迁移功能
    • Apache ShardingSphere：支持在线扩容

# 分表分库实战案例

# 案例 1：电商订单系统

# 背景

某电商平台订单数据量达到 1 亿行，查询性能下降明显。

# 解决方案

  1. 水平分库：按订单 ID 哈希分 4 个库
  2. 水平分表：每个库按月份分表
  3. 热点数据处理：最近 3 个月的订单保留在主库，历史订单迁移到冷存储

# 实现效果

  • 查询响应时间从 500ms 降至 50ms
  • 系统并发处理能力提升 5 倍
  • 数据备份和恢复时间缩短 80%

# 案例 2：社交应用用户系统

# 背景

用户量突破 1000 万，用户相关表数据量增长迅速。

# 解决方案

  1. 垂直分库：将用户核心信息和扩展信息分离
  2. 水平分表：用户表按用户 ID 哈希分 8 个表
  3. 缓存策略：热点用户数据缓存到 Redis

# 实现效果

  • 用户信息查询响应时间降至 10ms 以内
  • 系统支持用户量轻松扩展到 5000 万

# 分表分库工具

# 1. Apache ShardingSphere

  • 功能：提供完整的分库分表解决方案，支持数据分片、读写分离、分布式事务等
  • 优点：功能强大，生态成熟，支持多种数据库
  • 适用场景：中大型应用，需要完整的分库分表功能

# 2. MyBatis-Plus Sharding

  • 功能：基于 MyBatis-Plus 的分库分表插件
  • 优点：集成简单，与 MyBatis 无缝对接
  • 适用场景：使用 MyBatis 的项目，轻量级分库分表需求

# 3. Canal

  • 功能：基于 MySQL binlog 的增量数据同步工具
  • 优点：实时数据同步，支持多种数据源
  • 适用场景：数据迁移、异构系统数据同步

# 4. Vitess

  • 功能：Google 开源的数据库分片系统
  • 优点：支持水平扩展，高可用性
  • 适用场景：大规模分布式系统

# 总结

分表分库是解决数据库性能瓶颈的有效手段，但也带来了一定的复杂性。在实施分表分库时，需要考虑以下因素：

  1. 业务需求：根据业务特点选择合适的分片策略
  2. 数据分布：确保数据均匀分布，避免热点问题
  3. 查询模式：优化查询方式，减少跨分片操作
  4. 扩容方案：设计灵活的扩容策略，支持业务增长
  5. 运维成本：考虑分表分库后的运维复杂度

通过合理的分表分库设计，可以显著提升系统性能和可扩展性，为业务发展提供有力支持。
