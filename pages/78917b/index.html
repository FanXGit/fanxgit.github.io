    # MySQL - 存储引擎

    =============================

    本文主要介绍MySQL中的存储引擎。

    • MySQL - 存储引擎
      • InnoDB
        • MVCC是什么
      • MyISAM
      • 比较

    # InnoDB

    InnoDB是 MySQL 默认的事务型存储引擎，只有在需要它不支持的特性时，才考虑使用其它存储引擎。

    Innodb 不支持的特性场景:

    特性场景 具体说明 具备该特性的存储引擎
    纯内存存储 数据全部存放于内存，读写极快，重启 / 断电数据丢失 Memory（Heap）
    极致压缩归档 只支持 INSERT/SELECT，极高压缩比，适合日志归档 Archive
    表级锁设计 仅支持表锁，无行锁，不支持事务并发 MyISAM
    极速 COUNT (*) 引擎内置总行数，无需扫描即可快速返回 MyISAM
    延迟更新索引 延迟写入索引提升写入速度，崩溃易索引损坏 MyISAM
    堆表存储结构 数据与索引分离，非聚簇索引组织 MyISAM
    早期全文 / 空间索引 MySQL 5.6 前仅支持 FULLTEXT/SPATIAL 索引 MyISAM
    数据文件直接拷贝 表文件独立，可直接复制迁移，无需表空间管理 MyISAM
    无事务、无回滚 不支持事务、回滚、外键，结构简单开销低 MyISAM、Memory、Archive

    采用 MVCC 来支持高并发，并且实现了四个标准的隔离级别，默认级别是可重复读(REPEATABLE READ)。

    MySQL InnoDB 引擎的 4 个隔离级别，从低到高依次是：

    1. 读未提交:一个事务能读到其他事务未提交的数据
      • 读到别的事务未提交的数据，对方回滚后数据无效，就会导致 脏读 的问题。基本不用，数据一致性极差
    2. 读已提交:一个事务只能读到其他事务已提交的数据
      • 同一事务内，两次查询同一行，结果不一样（被别的事务 update/delete 并提交）,会导致不可重复读的问题。
    3. 可重复读: 一个事务内多次读取同一数据，结果始终一致(InnoDB 默认隔离级别)
      • 同一事务内，两次范围查询，结果集条数不一样（别的事务 insert 新行并提交）。会导致 幻读（Phantom Read）的问题。
    4. 串行化:最高级别，事务完全串行执行。
      • 每个事务自动加锁，无并发,极少使用
    隔离级别 脏读 不可重复读 幻读
    读未提交 ✅ 可能 ✅ 可能 ✅ 可能
    读已提交 ❌ 解决 ✅ 可能 ✅ 可能
    可重复读 ❌ 解决 ❌ 解决 InnoDB 下解决
    串行化 ❌解决 ❌ 解决 ❌ 解决

    可重复读到底会不会产生幻读？ MySQL 5.0 是分水岭：5.0 之前 InnoDB RR 有幻读；5.0 及之后通过 Next‑Key Lock + MVCC 彻底解决。 后续版本（5.7/8.0）仅优化加锁范围，核心防幻读机制未变。

    # MVCC

    MVCC = Multi-Version Concurrency Control (多版本并发控制)。

    核心思想：读写不互斥，读不加锁，写不阻塞读，极大提升并发。

    MVCC 核心原理: InnoDB 对每行数据，会保存多个历史版本。读操作：读取历史版本，不加锁,写操作：生成新版本，不阻塞旧版本读取.

    InnoDB 中 MVCC 怎么实现:

    1. 每行数据的 3 个隐藏字段,InnoDB 会给每行数据偷偷加 3 个字段：
      • DB_TRX_ID 生成该版本的事务 ID（自增数字）
      • DB_ROLL_PTR 回滚指针，指向 undo log 里的历史版本链
      • DB_ROW_ID 隐藏主键（表没主键时才用）
    2. undo log 版本链：每次 update/delete 不直接删旧数据，而是：
      • 生成一条新记录
      • 旧记录通过 DB_ROLL_PTR 链到 undo log形成一条版本链：
    最新版本 → 旧版本1 → 旧版本2 → ... → 最初版本
    1
    1. Read View（一致性视图）
      • 事务执行时生成一个 “快照”，用来判断：当前能看见版本链上的哪个版本。Read View 里 4 个关键字段：
        • m_ids：当前活跃的读写事务 ID 集合
        • min_trx_id：最小活跃事务 ID
        • max_trx_id：下一个要分配的事务 ID
        • creator_trx_id：当前事务自己的 ID
    2. 可见性判断规则
      • 版本的 trx_id == 自己事务 ID → 可见
      • 版本的 trx_id < min_trx_id → 已提交，可见
      • 版本的 trx_id >= max_trx_id → 不可见
      • 版本的 trx_id 在 m_ids 里 → 还在活跃，不可见
      • 否则 → 可见
    3. 不同隔离级别的区别
      • READ COMMITTED（RC） 每次 SELECT 都生成新 Read View
      • REPEATABLE READ（RR，默认） 事务中第一次 SELECT 生成一次，后面全程复用 所以事务内数据永远一致，解决不可重复读
    4. MVCC 只针对 “快照读”
      • 普通 SELECT：快照读，走 MVCC，不加锁
      • SELECT ... FOR UPDATE / UPDATE / DELETE：当前读，加锁，不走 MVCC

    # Next-Key Lock（间隙锁）是什么？

    Next-Key Lock = 记录锁（行锁） + 间隙锁（Gap Lock）

    作用:锁住某行 + 该行前后的间隙，不让别的事务在间隙里 insert。

    # mysql中各类log的作用

    日志名称 英文名称 作用 核心特点 属于谁
    重做日志 Redo Log 保障事务持久性，崩溃恢复，防止数据丢失 循环写，固定大小，WAL 核心，崩溃重启自动恢复 InnoDB
    回滚日志 Undo Log 实现事务回滚、MVCC 多版本、原子性 记录数据旧版本，用于回滚 & 快照读 InnoDB
    二进制日志 Binlog 主从复制、数据恢复、增量备份 记录逻辑修改（SQL / 行变化），追加写 MySQL Server
    错误日志 Error Log 记录启动、运行、崩溃等错误信息 排查问题最重要日志 MySQL Server
    查询日志 General Log 记录所有客户端连接和执行的 SQL 非常占空间，默认关闭 MySQL Server
    慢查询日志 Slow Query Log 记录执行超过阈值的慢 SQL 优化 SQL 必备，默认关闭 MySQL Server
    中继日志 Relay Log 从库同步主库 binlog，异步重放 只在从库存在 MySQL 主从复制
    DDL 日志 DDL Log 记录 DDL 操作，防止崩溃导致元数据不一致 保障 DDL 原子性 MySQL Server

    Innodb 主索引是聚簇索引，在索引中保存了数据，从而避免直接读取磁盘，因此对查询性能有很大的提升。

    内部做了很多优化，包括从磁盘读取数据时采用的可预测性读、能够加快读操作并且自动创建的自适应哈希索引、能够加速插入操作的插入缓冲区等。

    支持真正的在线热备份。其它存储引擎不支持在线热备份，要获取一致性视图需要停止对所有表的写入，而在读写混合场景中，停止写入可能也意味着停止读取。

    # MyISAM

    设计简单，数据以紧密格式存储。对于只读数据，或者表比较小、可以容忍修复操作，则依然可以使用它。

    提供了大量的特性，包括压缩表、空间数据索引等。

    不支持事务。

    不支持行级锁，只能对整张表加锁，读取时会对需要读到的所有表加共享锁，写入时则对表加排它锁。但在表有读取操作的同时，也可以往表中插入新的记录，这被称为并发插入(CONCURRENT INSERT)。

    可以手工或者自动执行检查和修复操作，但是和事务恢复以及崩溃恢复不同，可能导致一些数据丢失，而且修复操作是非常慢的。

    如果指定了 DELAY_KEY_WRITE 选项，在每次修改执行完成时，不会立即将修改的索引数据写入磁盘，而是会写到内存中的键缓冲区，只有在清理键缓冲区或者关闭表的时候才会将对应的索引块写入磁盘。这种方式可以极大的提升写入性能，但是在数据库或者主机崩溃时会造成索引损坏，需要执行修复操作。

    # 比较

    • 事务: InnoDB 是事务型的，可以使用 Commit 和 Rollback 语句。

    • 并发: MyISAM 只支持表级锁，而 InnoDB 还支持行级锁。

    • 外键: InnoDB 支持外键。

    • 备份: InnoDB 支持在线热备份。

    • 崩溃恢复: MyISAM 崩溃后发生损坏的概率比 InnoDB 高很多，而且恢复的速度也更慢。

    • 其它特性: MyISAM 支持压缩表和空间数据索引。)

    # MySQL（InnoDB为主）锁类型完整对照表

    分类维度 锁类型 英文名称 核心作用 关键特点/使用场景 支持引擎
    按锁粒度 行级锁 Row Lock 锁定单条数据行 并发高、冲突小；InnoDB核心锁，用于精准控制数据 InnoDB
    表级锁 Table Lock 锁定整张数据表 并发低、冲突大；手动锁表（LOCK TABLES）或MyISAM默认锁 MyISAM、InnoDB
    页级锁 Page Lock 锁定16KB数据页 粒度介于行锁与表锁之间，现在基本不用 BDB等（主流不支持）
    InnoDB标准锁（高频） 记录锁 Record Lock 锁定某一行的索引记录 真正的行锁，精准锁定单行（如where id=10），防止修改/删除 InnoDB
    间隙锁 Gap Lock 锁定索引之间的间隙 不锁记录本身，仅阻止间隙插入，用于RR级别防止幻读 InnoDB
    临键锁 Next-Key Lock 锁定记录+间隙 Record Lock + Gap Lock，InnoDB RR级别默认锁，彻底解决幻读 InnoDB
    插入意向锁 Insert Intention Lock 插入前声明意向 插入数据前加锁，多个事务插入不同间隙可共存，提升并发 InnoDB
    按共享/排他 共享锁 Shared Lock（S锁） 读锁，允许并发读 多个事务可同时加S锁；加S锁后不能加X锁；手动触发（LOCK IN SHARE MODE） InnoDB、MyISAM
    排他锁 Exclusive Lock（X锁） 写锁，独占资源 互斥锁，增删改默认加X锁；手动触发（FOR UPDATE）；加X锁后禁止其他锁 InnoDB、MyISAM
    按加锁方式 隐式锁 Implicit Lock 自动加锁，无需手动操作 InnoDB增删改自动加X锁，普通SELECT无锁（快照读） InnoDB
    显式锁 Explicit Lock 手动指定加锁 FOR UPDATE、LOCK IN SHARE MODE（行锁）；LOCK TABLES（表锁） InnoDB、MyISAM
    特殊锁/机制 意向共享锁 Intention Shared Lock（IS锁） 表级标识，声明行锁意向 表级锁，事务准备给行加S锁时自动加；快速判断表中是否有行锁 InnoDB
    意向排他锁 Intention Exclusive Lock（IX锁） 表级标识，声明行锁意向 表级锁，事务准备给行加X锁时自动加；与其他IX/IS锁兼容，与X锁互斥 InnoDB
    自增锁 AUTO-INC Lock 保证自增ID连续 插入自增列时的特殊表锁；新版InnoDB优化为轻量级锁，提升并发 InnoDB
    谓词锁 Predicate Lock 锁定满足条件的范围数据 用于空间索引（GIS），防止范围查询时的幻读 InnoDB
