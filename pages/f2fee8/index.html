MySQL - 主从复制与读写分离

# MySQL - 主从复制与读写分离

=======================================

# 主从复制

主从复制是 MySQL 高可用性和扩展性的基础，通过将主服务器的数据复制到从服务器，实现数据备份和负载均衡。

# 复制原理

主从复制主要涉及三个线程:

  • binlog 线程 : 负责将主服务器上的数据更改写入二进制日志中。
  • I/O 线程 : 负责从主服务器上读取二进制日志，并写入从服务器的中继日志中。
  • SQL 线程 : 负责读取中继日志并重放其中的 SQL 语句。

# 复制类型

# 1. 基于语句的复制 (SBR)

  • 原理：主服务器记录 SQL 语句，从服务器执行相同的 SQL 语句
  • 优点：二进制日志小，复制效率高
  • 缺点：某些语句（如 NOW()、RAND()）可能导致数据不一致

# 2. 基于行的复制 (RBR)

  • 原理：主服务器记录数据行的变更，从服务器直接应用这些变更
  • 优点：数据一致性更好，不受 SQL 语句影响
  • 缺点：二进制日志较大，复制效率相对较低

# 3. 混合复制 (MBR)

  • 原理：默认使用基于语句的复制，对于可能导致不一致的语句自动切换为基于行的复制
  • 优点：兼顾效率和一致性
  • 缺点：实现复杂度较高

# 主从复制配置步骤

# 1. 主服务器配置

-- 1. 编辑 my.cnf 文件
[mysqld]
server-id = 1           # 服务器唯一 ID
log-bin = mysql-bin     # 启用二进制日志
binlog-format = ROW     # 二进制日志格式

-- 2. 创建复制用户
CREATE USER 'repl'@'%' IDENTIFIED BY 'password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';

-- 3. 查看主服务器状态
SHOW MASTER STATUS;
-- 记录 File 和 Position 值，用于从服务器配置

# 2. 从服务器配置

-- 1. 编辑 my.cnf 文件
[mysqld]
server-id = 2           # 服务器唯一 ID，不能与主服务器相同
relay-log = relay-bin   # 中继日志
read-only = 1           # 设置为只读模式

-- 2. 配置主从复制
CHANGE MASTER TO
    MASTER_HOST = 'master_ip',
    MASTER_USER = 'repl',
    MASTER_PASSWORD = 'password',
    MASTER_LOG_FILE = 'mysql-bin.000001',  # 主服务器的 File 值
    MASTER_LOG_POS = 12345;                # 主服务器的 Position 值

-- 3. 启动从服务器复制线程
START SLAVE;

-- 4. 查看从服务器状态
SHOW SLAVE STATUS\G;
-- 确保 Slave_IO_Running 和 Slave_SQL_Running 都为 Yes

# 主从复制常见问题及解决方案

# 1. 复制延迟

  • 原因：网络延迟、从服务器性能不足、主服务器写入量过大
  • 解决方案：
    • 优化网络连接
    • 提升从服务器硬件配置
    • 调整从服务器参数（如 slave_parallel_workers）
    • 使用并行复制

# 2. 复制错误

  • 原因：主键冲突、表结构不一致、权限问题
  • 解决方案：
    • 跳过错误（临时解决方案）：SET GLOBAL sql_slave_skip_counter = 1; START SLAVE;
    • 重新初始化从服务器
    • 确保主从服务器表结构一致

# 3. 主服务器故障

  • 解决方案：
    • 提升从服务器为主服务器
    • 更新应用连接配置
    • 重新配置其他从服务器指向新的主服务器

# 读写分离

读写分离是在主从复制的基础上，将写操作发送到主服务器，读操作分发到从服务器，从而提高系统的并发处理能力。

# 读写分离的优势

  • 缓解锁争用：主从服务器负责各自的读和写，减少锁冲突
  • 提高查询性能：从服务器可以使用 MyISAM 引擎，适合读操作
  • 增加冗余：提高系统可用性，主服务器故障时可切换到从服务器
  • 负载均衡：分散读请求，提高系统整体性能

# 读写分离实现方案

# 1. 应用层实现

在应用代码中根据 SQL 类型判断，将写操作发送到主服务器，读操作发送到从服务器。

优点：实现简单，不需要额外组件 缺点：代码耦合度高，维护成本大

示例代码：

// Java 代码示例
public class DataSourceRouter {
    private DataSource masterDataSource;
    private List<DataSource> slaveDataSources;
    
    public Connection getConnection(boolean isWrite) {
        if (isWrite) {
            return masterDataSource.getConnection();
        } else {
            // 随机选择一个从服务器
            int index = new Random().nextInt(slaveDataSources.size());
            return slaveDataSources.get(index).getConnection();
        }
    }
}

# 2. 代理层实现

使用专门的代理服务器（如 MySQL Proxy、ProxySQL、MaxScale）来实现读写分离。

优点：应用代码无需修改，集中管理连接 缺点：增加了系统复杂度和单点故障风险

# 常用读写分离工具

# 1. MySQL Proxy

  • 特点：MySQL 官方提供的代理工具，基于 Lua 脚本
  • 优点：开源，轻量级
  • 缺点：性能一般，配置复杂

# 2. ProxySQL

  • 特点：高性能的 MySQL 代理，支持读写分离、连接池、查询路由
  • 优点：性能优异，功能丰富，支持监控
  • 缺点：配置相对复杂

# 3. MaxScale

  • 特点：MariaDB 提供的代理工具，支持读写分离、故障转移
  • 优点：稳定可靠，功能完善
  • 缺点：资源占用较高

# 4. MyCat

  • 特点：国产开源分布式数据库中间件，支持读写分离、分库分表
  • 优点：功能全面，易于使用
  • 缺点：性能一般

# 读写分离最佳实践

# 1. 数据一致性处理

  • 延迟问题：从服务器存在复制延迟，可能读取到旧数据
  • 解决方案：
    • 对实时性要求高的查询直接访问主服务器
    • 使用半同步复制减少复制延迟
    • 实现数据版本控制，确保读取最新数据

# 2. 连接池管理

  • 主服务器：使用较小的连接池，避免过多连接影响写性能
  • 从服务器：使用较大的连接池，应对高并发读请求

# 3. 监控与告警

  • 复制状态监控：定期检查从服务器复制状态
  • 延迟监控：监控主从复制延迟，及时发现问题
  • 性能监控：监控各服务器的负载情况，及时调整配置

# 主从复制与高可用

# 1. 半同步复制

  • 原理：主服务器在提交事务前，至少等待一个从服务器确认接收了二进制日志
  • 优点：减少数据丢失的风险
  • 缺点：可能增加主服务器的响应时间

配置：

-- 主服务器
SET GLOBAL rpl_semi_sync_master_enabled = 1;
SET GLOBAL rpl_semi_sync_master_timeout = 10000; -- 10秒超时

-- 从服务器
SET GLOBAL rpl_semi_sync_slave_enabled = 1;

# 2. 级联复制

  • 原理：从服务器同时作为其他从服务器的主服务器
  • 优点：减轻主服务器的复制压力
  • 适用场景：从服务器数量较多的情况

# 3. 故障转移

  • 手动故障转移：手动将从服务器提升为主服务器
  • 自动故障转移：使用工具（如 MHA、Orchestrator）实现自动故障检测和转移

# MHA (Master High Availability)

  • 功能：自动监控主服务器状态，主服务器故障时自动将从服务器提升为主服务器
  • 优点：实现自动故障转移，减少 downtime
  • 缺点：配置复杂，需要额外的管理节点

# 实战案例

# 案例：电商系统读写分离架构

# 架构设计

  • 主服务器：处理所有写操作和实时性要求高的读操作
  • 从服务器集群：3 台从服务器，处理普通读操作
  • ProxySQL：作为中间代理，实现读写分离和负载均衡
  • 监控系统：使用 Prometheus + Grafana 监控复制状态和性能

# 实现效果

  • 性能提升：读请求响应时间从 200ms 降至 50ms
  • 高可用性：主服务器故障时，自动切换到从服务器， downtime 控制在 30 秒以内
  • 可扩展性：支持水平扩展从服务器数量，应对业务增长

# 总结

主从复制与读写分离是 MySQL 高可用和高性能架构的重要组成部分：

  1. 主从复制：提供数据备份、故障恢复和负载均衡的基础
  2. 读写分离：通过分散读写请求，提高系统并发处理能力
  3. 高可用方案：结合半同步复制、故障转移等技术，确保系统稳定性
  4. 监控与维护：建立完善的监控体系，及时发现和解决问题

通过合理配置和管理主从复制与读写分离，可以显著提升 MySQL 数据库的性能、可用性和可扩展性，为业务发展提供有力支持。
