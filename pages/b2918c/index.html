# 准备知识点

# 什么是幂等？

幂等原先是数学中的一个概念，表示进行1次变换和进行N次变换产生的效果相同。

当我们讨论接口的幂等性时，一般是指：以相同的请求调用这个接口一次和调用这个接口多次，对系统产生的影响是相同的。如果一个接口满足这个特性，那么我们就说这个接口是一个幂等接口。

# 什么是接口幂等？

在HTTP/1.1中，对幂等性进行了定义。它描述了一次和多次请求某一个资源对于资源本身应该具有同样的结果（网络超时等问题除外），即第一次请求的时候对资源产生了副作用，但是以后的多次请求都不会再对资源产生副作用。

这里的副作用是指不会对结果产生破坏或者产生不可预料的结果。也就是说，其任意多次执行对资源本身所产生的影响均与一次执行的影响相同。

# 哪些接口需要保证幂等？

我们来看标准的RESTful请求的幂等性情况：

HTTP方法 幂等性 说明
GET 幂等 只是获取资源，对资源本身没有任何副作用
HEAD 幂等 本质上和GET一样，获取头信息
OPTIONS 幂等 获取当前URL所支持的方法
DELETE 幂等 删除操作，多次删除结果相同
PUT 幂等 用于更新资源，多次更新结果相同
POST 非幂等 用于添加资源，多次提交可能产生多笔记录

需要保证幂等的接口主要是：

  1. POST请求：用于创建资源的接口，如订单提交、支付等
  2. 部分PUT请求：某些更新操作可能需要保证幂等
  3. DELETE请求：虽然理论上是幂等的，但实际实现中可能需要处理

# 常见的保证幂等的方式

# 数据库层面

# 悲观锁

实现方式：使用数据库的 for update 语句

select * from t_order where order_id = trade_no for update;
1

工作原理：

  1. 当线程A执行for update时，数据会对当前记录加锁
  2. 其他线程执行到此行代码时，会等待线程A释放锁
  3. 事务提交时，for update获取的锁会自动释放

优缺点：

  • 优点：实现简单，可靠性高
  • 缺点：并发性能差，线程会长期处于等待状态

# 唯一ID/索引

实现方式：利用数据库唯一约束特性

工作原理：

  1. 为业务唯一标识创建唯一索引
  2. 插入数据时，如果重复会触发唯一约束异常
  3. 捕获异常并处理，实现幂等

适用场景：插入操作，如订单创建、支付记录等

去重表：

  • 专门创建一张去重表，将业务唯一标识作为唯一索引
  • 与主业务操作放在同一事务中
  • 重复操作时，数据库会抛出唯一约束异常，事务回滚

# 乐观锁（基于版本号或者时间戳）

实现方式：在表中添加版本号或时间戳字段

工作原理：

  1. 查询数据时获取当前版本号
  2. 更新时带上版本号条件
  3. 只有版本号匹配时才执行更新

代码示例：

update goods set name=#{newName}, version=#{version+1} where id=#{id} and version=#{version}
1

状态机：

  • 利用业务状态的流转来保证幂等
  • 例如：订单状态从创建(0) → 付款(100)，更新时添加状态条件
update `order` set status=#{status} where id=#{id} and status<#{status}
1

# 分布式锁

实现方式：使用Redis、Zookeeper等实现分布式锁

工作原理：

  1. 生成业务唯一标识作为锁的key
  2. 尝试获取分布式锁
  3. 获取成功则执行业务逻辑，执行完成后释放锁
  4. 获取失败则直接返回，不执行业务逻辑

代码示例：

// 使用Redis实现分布式锁
public boolean acquireLock(String key, long expireTime) {
    return redisTemplate.opsForValue().setIfAbsent(key, "1", expireTime, TimeUnit.SECONDS);
}

public void releaseLock(String key) {
    redisTemplate.delete(key);
}

# Token机制

实现方式：基于Token的防重复提交机制

工作原理：

  1. 客户端请求获取Token
  2. 服务端生成Token并存储（如Redis）
  3. 客户端携带Token发起业务请求
  4. 服务端验证Token有效性
  5. 验证通过则执行业务逻辑，同时删除Token
  6. 验证失败则拒绝请求

适用场景：表单提交、支付请求等需要防止重复提交的场景

# 基于请求头的幂等性方案

实现方式：使用请求头中的唯一标识来保证幂等

工作原理：

  1. 客户端生成唯一标识（如UUID）
  2. 将唯一标识放入请求头（如X-Request-ID）
  3. 服务端检查请求头中的唯一标识
  4. 若该标识已处理过，则直接返回之前的结果
  5. 若未处理过，则执行业务逻辑并记录该标识

代码示例：

@RestController
@RequestMapping("/api")
public class ApiController {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    @PostMapping("/payment")
    public ResponseResult<Payment> processPayment(
            @RequestHeader("X-Request-ID") String requestId,
            @RequestBody PaymentDTO paymentDTO) {
        
        // 检查请求是否已处理
        if (redisTemplate.hasKey("payment:" + requestId)) {
            // 返回之前的结果
            String resultJson = redisTemplate.opsForValue().get("payment:" + requestId);
            return JSON.parseObject(resultJson, ResponseResult.class);
        }
        
        // 处理支付逻辑
        Payment payment = paymentService.process(paymentDTO);
        ResponseResult<Payment> result = ResponseResult.success(payment);
        
        // 缓存结果
        redisTemplate.opsForValue().set(
                "payment:" + requestId,
                JSON.toJSONString(result),
                24, TimeUnit.HOURS
        );
        
        return result;
    }
}

# 基于业务参数的幂等性方案

实现方式：使用业务参数组合生成唯一标识

工作原理：

  1. 根据业务参数组合生成唯一标识
  2. 使用该标识作为幂等键
  3. 检查该键是否已存在
  4. 不存在则执行业务逻辑并存储该键

代码示例：

@Service
public class OrderService {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    @Autowired
    private OrderRepository orderRepository;
    
    @Transactional
    public Order createOrder(OrderDTO orderDTO) {
        // 生成业务唯一标识
        String orderKey = "order:" + orderDTO.getUserId() + ":" + orderDTO.getProductId();
        
        // 尝试获取锁
        Boolean acquired = redisTemplate.opsForValue().setIfAbsent(
                orderKey, "processing", 30, TimeUnit.SECONDS
        );
        
        if (!acquired) {
            throw new BusinessException("订单正在处理中");
        }
        
        try {
            // 检查是否已存在订单
            Order existingOrder = orderRepository.findByUserIdAndProductId(
                    orderDTO.getUserId(), orderDTO.getProductId()
            );
            
            if (existingOrder != null) {
                return existingOrder;
            }
            
            // 创建新订单
            Order order = new Order();
            order.setUserId(orderDTO.getUserId());
            order.setProductId(orderDTO.getProductId());
            order.setQuantity(orderDTO.getQuantity());
            order.setStatus(OrderStatus.CREATED);
            
            return orderRepository.save(order);
        } finally {
            // 释放锁
            redisTemplate.delete(orderKey);
        }
    }
}

# 消息队列的幂等性处理

实现方式：使用消息ID或业务唯一标识保证幂等

工作原理：

  1. 消息生产者生成唯一消息ID
  2. 消息消费者消费消息前检查是否已处理
  3. 未处理则消费并记录已处理标识
  4. 已处理则直接跳过

代码示例：

@Component
public class OrderConsumer {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    @Autowired
    private OrderService orderService;
    
    @RabbitListener(queues = "order.queue")
    public void processOrder(Message message, Channel channel) throws IOException {
        String messageId = message.getMessageProperties().getMessageId();
        
        // 检查消息是否已处理
        if (redisTemplate.hasKey("message:" + messageId)) {
            // 确认消息
            channel.basicAck(message.getMessageProperties().getDeliveryTag(), false);
            return;
        }
        
        try {
            // 处理订单
            String orderJson = new String(message.getBody(), StandardCharsets.UTF_8);
            OrderDTO orderDTO = JSON.parseObject(orderJson, OrderDTO.class);
            orderService.createOrder(orderDTO);
            
            // 标记消息已处理
            redisTemplate.opsForValue().set(
                    "message:" + messageId, "processed", 
                    7, TimeUnit.DAYS
            );
            
            // 确认消息
            channel.basicAck(message.getMessageProperties().getDeliveryTag(), false);
        } catch (Exception e) {
            // 拒绝消息
            channel.basicNack(message.getMessageProperties().getDeliveryTag(), false, false);
        }
    }
}

# 缓存层面的幂等性处理

实现方式：使用缓存存储请求结果

工作原理：

  1. 计算请求的唯一键（如URL+参数的哈希值）
  2. 检查缓存中是否存在该键的结果
  3. 存在则直接返回缓存结果
  4. 不存在则执行业务逻辑并缓存结果

代码示例：

@RestController
@RequestMapping("/api")
public class ProductController {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    @Autowired
    private ProductService productService;
    
    @GetMapping("/product/{id}")
    public ResponseResult<Product> getProduct(@PathVariable Long id) {
        // 生成缓存键
        String cacheKey = "product:" + id;
        
        // 检查缓存
        String cachedResult = redisTemplate.opsForValue().get(cacheKey);
        if (cachedResult != null) {
            return JSON.parseObject(cachedResult, ResponseResult.class);
        }
        
        // 查询产品
        Product product = productService.findById(id);
        ResponseResult<Product> result = ResponseResult.success(product);
        
        // 缓存结果
        redisTemplate.opsForValue().set(
                cacheKey, JSON.toJSONString(result),
                5, TimeUnit.MINUTES
        );
        
        return result;
    }
}

# Token机制实现案例

# 实现思路

  1. Token生成：提供一个接口用于生成Token，存储到Redis中，设置过期时间
  2. Token验证：提供一个注解用于标记需要幂等的接口
  3. 拦截器：拦截带有幂等注解的请求，验证Token有效性
  4. 异常处理：处理Token验证失败的情况

# 3. Token生成工具

@Component
public class TokenUtil {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    /**
     * 生成Token
     */
    public String generateToken() {
        String token = UUID.randomUUID().toString().replace("-", "");
        redisTemplate.opsForValue().set(token, token, 5, TimeUnit.MINUTES);
        return token;
    }
    
    /**
     * 验证Token
     */
    public boolean validateToken(String token) {
        if (StringUtils.isBlank(token)) {
            return false;
        }
        return redisTemplate.delete(token);
    }
}

# 4. 拦截器

public class IdempotentInterceptor implements HandlerInterceptor {
    
    @Autowired
    private TokenUtil tokenUtil;
    
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        if (handler instanceof HandlerMethod) {
            HandlerMethod handlerMethod = (HandlerMethod) handler;
            Idempotent idempotent = handlerMethod.getMethodAnnotation(Idempotent.class);
            if (idempotent != null) {
                String token = request.getParameter(idempotent.value());
                if (!tokenUtil.validateToken(token)) {
                    response.setStatus(HttpStatus.BAD_REQUEST.value());
                    response.setContentType(MediaType.APPLICATION_JSON_VALUE);
                    response.getWriter().write(JSON.toJSONString(ResponseResult.error("重复请求")));
                    return false;
                }
            }
        }
        return true;
    }
}

# 最佳实践

  1. 选择合适的幂等方案：

    • 插入操作：使用唯一ID/索引
    • 更新操作：使用乐观锁
    • 分布式场景：使用分布式锁
    • 表单提交：使用Token机制

  2. Token机制最佳实践：

    • Token设置合理的过期时间
    • Token存储在Redis中，使用原子操作
    • 拦截器统一处理Token验证
    • 提供清晰的错误提示

  3. 性能考虑：

    • 合理设置Redis缓存大小
    • 避免长时间占用分布式锁
    • 优化数据库索引

  4. 安全性：

    • Token要足够随机，避免被猜测
    • 敏感操作要增加额外的验证
    • 定期清理过期的Token

# 常见问题

# 1. Token过期怎么办？

解决方案：设置合理的过期时间，一般为5-10分钟。客户端在Token过期后重新获取。

# 2. 分布式环境下Token同步问题？

解决方案：使用Redis等分布式缓存存储Token，确保多实例共享。

# 3. 高并发下Token竞争问题？

解决方案：使用Redis的原子操作（如delete），确保Token只能被消费一次。

# 4. 前端如何处理Token？

解决方案：

  • 表单提交前获取Token
  • 将Token存储在表单隐藏字段中
  • 提交后禁用提交按钮
  • 刷新页面时重新获取Token

# 5. 如何处理网络重试？

解决方案：

  • 客户端实现幂等性，使用相同的Token进行重试
  • 服务端保证Token验证的原子性

# 幂等性方案汇总表

方案类型 实现方式 适用场景 优点 缺点
悲观锁 使用for update语句 数据库操作 实现简单，可靠性高 并发性能差，线程等待时间长
唯一ID/索引 利用数据库唯一约束 插入操作（订单、支付） 实现简单，可靠性高 需要设计合理的唯一标识
乐观锁 基于版本号或时间戳 更新操作 并发性能好，无锁竞争 需要额外字段，实现复杂度较高
分布式锁 Redis/Zookeeper实现 分布式场景 支持分布式环境 实现复杂度高，依赖外部组件
Token机制 生成唯一Token并验证 表单提交、支付请求 实现灵活，适用范围广 需要额外的Token管理
请求头方案 使用X-Request-ID等请求头 API调用、第三方接口集成 无侵入性，实现简单 依赖客户端正确传递请求头
业务参数方案 基于业务参数生成唯一标识 订单创建、支付处理 与业务紧密结合 需要合理设计业务唯一标识
消息队列方案 使用消息ID或业务唯一标识 消息队列消费、异步任务 确保消息只被消费一次 实现复杂度较高，依赖消息系统
缓存方案 缓存请求结果 查询接口、数据获取 性能优异，减少重复计算 缓存一致性问题，过期时间管理

# 方案选择指南

业务场景 推荐方案 原因
订单创建 唯一ID/索引 + 业务参数方案 确保订单唯一性，避免重复创建
支付处理 Token机制 + 唯一ID/索引 防止重复支付，确保交易安全
API调用 请求头方案 无侵入性，易于集成
数据更新 乐观锁 并发性能好，适合高频更新
分布式任务 分布式锁 确保任务只被执行一次
消息处理 消息队列方案 确保消息只被消费一次
查询接口 缓存方案 提高性能，减少数据库压力
表单提交 Token机制 防止重复提交，提升用户体验

通过本文的学习，你应该对接口幂等性有了全面的了解，并掌握了多种实现方案。在实际开发中，应根据具体业务场景选择合适的幂等方案，确保系统的可靠性和稳定性。

选择幂等方案时，需要考虑以下因素：

  1. 业务场景：不同场景适合不同的幂等方案
  2. 性能要求：高并发场景需要选择性能更好的方案
  3. 实现复杂度：根据团队技术水平选择合适的方案
  4. 可靠性：关键业务需要选择可靠性更高的方案
  5. 维护成本：考虑方案的长期维护成本

综合考虑这些因素，选择最适合当前业务场景的幂等方案，才能真正保证系统的稳定性和可靠性。
